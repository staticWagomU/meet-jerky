# 設計上の注意点

- `sanitize_pcm_sample` と `validate_audio_format_properties` はどちらも `pub` ではない private fn だが、同モジュールの `mod tests` 内から呼べる (`use super::*` で既存のとおり)。
- `validate_audio_format_properties` は `#[cfg(target_os = "macos")]` ガードが付いているか? 確認: L82 のソースを見る限り、`fn validate_audio_format_properties(...)` 単体には cfg なし、`fn validate_audio_format_description(...)` のみ macOS gate (L113)。よって test は cfg なしで OK のはず。読んで確認すること。
- F/G の入力の `Some(1)` / `Some(32)` は最後に判定される条件が None で先に返るようにするため。
- H の `assert_eq!` で `Err(&'static str)` を比較するときは `result.unwrap_err()` で `&'static str` を取り出して比較する書き方が自然。`Result<(), &'static str>` なら `assert_eq!(result.unwrap_err(), "...")`。
- 既存テストでは `assert!(...is_err())` のみで文言を確認していない。Hには「既存 5 件は文言確認していない、本 test がそれを補強」というコメントを test 関数定義の前に 1 行で残してよい。
