# 1. 背景 / Why

`normalize_speaker` (line 98-103) は話者ラベルを正規化する private fn。実装は以下の 1 行ロジック:

```rust
fn normalize_speaker(raw: Option<&str>) -> String {
    match raw.map(str::trim) {
        Some(s) if !s.is_empty() => s.to_string(),
        _ => "不明".to_string(),
    }
}
```

既存 5 件 (line 301-327) は以下を確認済:
- `None` → "不明"
- `Some("")` → "不明"
- `Some("   ")` (ASCII spaces) / `Some("\t\n  ")` (ASCII tab/newline/spaces) → "不明"
- `Some("Alice")` / `Some("自分")` / `Some("相手側")` → そのまま (trim 後 non-empty)
- `Some("  Alice  ")` / `Some(" Alice Bob ")` → 前後 ASCII whitespace trim

**確認した gap**:
1. **U+3000 全角空白**: `str::trim` は Unicode `White_Space` プロパティ全体を対象にするので U+3000 もトリム対象。既存テストは ASCII whitespace のみで、U+3000 trim の現契約は未確認。
2. **U+3000 のみ入力での fallback**: Unicode whitespace のみで構成された input は trim 後 empty になり "不明" にフォールバックする現契約も未確認。
3. **NUL byte (`\0`)**: NUL は `char::is_whitespace` で false なので trim 対象外、passthrough される現契約。
4. **Control character (`\x01` 等)**: ASCII control characters の多くは whitespace ではないので trim されず passthrough される現契約。
5. **Zero-width space (ZWSP, U+200B)**: ZWSP は Unicode 上「不可視だが whitespace ではない」現契約 (`char::is_whitespace` で false)。trim されず passthrough。

これら 5 件で `str::trim` の Unicode whitespace 解釈境界を完全に固定する。`is_whitespace` 仕様の解釈変更や、独自 trim 実装への誤リファクタを CI で検知する装置になる。

# 2. 追加する 5 件のテスト案 (assertion レベル)

すべて `transcript_bridge.rs` の `mod tests` 末尾に追加。private fn `normalize_speaker` は同じファイル内の `tests` mod から `super::*` 経由で直接呼べる (既存パターン踏襲、line 301-327 と同じ呼び方)。

## T1. `normalize_speaker_trims_unicode_full_width_space_u3000`
- 入力: `Some("\u{3000}相手側\u{3000}")` (U+3000 全角空白で挟まれた CJK)
- assert: `assert_eq!(normalize_speaker(Some("\u{3000}相手側\u{3000}")), "相手側")`
- 補助 assert (混在パターン): `assert_eq!(normalize_speaker(Some("\u{3000}  Alice  \u{3000}")), "Alice")` (U+3000 + ASCII spaces 混在も trim される)
- 意図: `str::trim` が Unicode `White_Space` プロパティ全体を対象にする現契約を固定。ASCII whitespace のみ trim する独自実装への誤リファクタを検知。

## T2. `normalize_speaker_returns_unknown_for_only_unicode_full_width_spaces`
- 入力: `Some("\u{3000}\u{3000}")` (U+3000 のみ)
- assert: `assert_eq!(normalize_speaker(Some("\u{3000}\u{3000}")), "不明")`
- 補助 assert (Unicode whitespace 混在のみ): `assert_eq!(normalize_speaker(Some("\u{3000}\t\n\u{3000}")), "不明")` (U+3000 + ASCII whitespace 混在で構成された input も trim 後 empty)
- 意図: trim 後 empty の判定が Unicode whitespace でも正しく fallback "不明" に倒れる現契約を固定。`is_empty()` 判定漏れを検知。

## T3. `normalize_speaker_passes_through_nul_byte_label`
- 入力: `Some("\0")` (単一 NUL byte)
- assert: `assert_eq!(normalize_speaker(Some("\0")), "\0")` (passthrough)
- 補助 assert (NUL + 通常文字): `assert_eq!(normalize_speaker(Some("\0Alice")), "\0Alice")` (NUL は前置でも trim されない)
- 意図: NUL byte が `char::is_whitespace` で false である現契約を固定。NUL を whitespace 扱いする独自実装への誤リファクタを検知。

## T4. `normalize_speaker_passes_through_control_character_label`
- 入力: `Some("\x01alpha")` (SOH = U+0001 + 通常文字)
- assert: `assert_eq!(normalize_speaker(Some("\x01alpha")), "\x01alpha")` (passthrough、`\x01` は trim 対象外)
- 補助 assert (DEL = U+007F): `assert_eq!(normalize_speaker(Some("\u{007F}beta")), "\u{007F}beta")` (DEL も control だが whitespace ではない、passthrough)
- 意図: ASCII control characters の多くが `char::is_whitespace` で false である現契約を固定 (whitespace 扱いされる例外は `\t \n \r \x0B \x0C` の 5 種のみ)。control character を一律 trim する独自実装への誤リファクタを検知。

## T5. `normalize_speaker_passes_through_zero_width_space_label`
- 入力: `Some("\u{200B}name")` (ZWSP + 通常文字)
- assert: `assert_eq!(normalize_speaker(Some("\u{200B}name")), "\u{200B}name")` (passthrough、ZWSP は trim 対象外)
- 補助 assert (ZWJ = U+200D): `assert_eq!(normalize_speaker(Some("\u{200D}name")), "\u{200D}name")` (ZWJ も passthrough)
- 補助 assert (ZWSP のみ input): `assert_eq!(normalize_speaker(Some("\u{200B}\u{200B}")), "\u{200B}\u{200B}")` (ZWSP のみでも trim されないので fallback ではなく passthrough)
- 意図: Zero-width space / Zero-width joiner が「不可視だが whitespace ではない」現契約 (`char::is_whitespace` で false) を固定。Unicode 上の zero-width 文字を whitespace 扱いする独自実装への誤リファクタを検知。

# 3. 実装上の注意

- 既存テスト 5 件 (line 301-327) は `super::*` import 経由で `normalize_speaker` を直接呼んでいる。同じパターンを踏襲する。
- 関数本体は **絶対に変更しない**。テスト追加のみ。
- Unicode escape (`\u{3000}` / `\u{200B}` / `\u{200D}` / `\u{007F}` / `\x01` / `\0`) は Rust の string literal でそのまま使える。
- 補助 assert を 1 件の `#[test]` 関数内に複数並べてよい (既存 `normalize_speaker_returns_unknown_for_whitespace_only` の 2 assert パターン踏襲)。
- assert マクロは `assert_eq!` のみ使う、`#[should_panic]` は不要。
- 既存テストとの重複を避けるため、ASCII whitespace のみの input は新規テストでは扱わない (T2 の補助 assert で「U+3000 + ASCII whitespace 混在」を入れるのは既存とは異なるシナリオなので OK)。
- inline コメントは「現契約を意図的に固定」が読み取れる 1 行で十分 (例: `// U+3000 は str::trim の対象`)。長文コメント不要。
