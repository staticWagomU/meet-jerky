# 背景 / Why

直前のループ (mjc-main-15 Loop 1, commit 200f97f) で audio-drop-count chain (rust extract → frontend listen → UI display → session reset) の semantics を完成させました。

ただし、event 名 `"audio-drop-count"` は現在以下の **3 箇所に重複** しています:

- `src-tauri/src/audio.rs` line 401 (mic emitter スレッド)
- `src-tauri/src/system_audio.rs` line 333 (system emitter スレッド、cfg(target_os = "macos") ガード内)
- `src/routes/TranscriptView.tsx` line 809 (frontend listener)

frontend (TypeScript) は別言語空間で同期は手動責任のため触りません。本ループでは **rust 側 2 箇所** を `audio_event::AUDIO_DROP_EVENT_NAME` 定数経由に集約します。

これは Kent Beck 流の Tidy First (構造変化のみ・振る舞い不変・テストの assertion 内容は変更しない) です。価値は:

1. **将来 event 名を変更する場合の単一書き換え地点化** (現在は rust 2 箇所同時編集が必要)
2. **「event 名は audio_event モジュールが所有する」という所有権境界の明確化** (前 mjc-main-14 Loop 1 で audio_event.rs を抽出した方針の延長)
3. **typo 検知** (文字列リテラル → 定数で compile error 化)

#[cfg(test)]
mod tests {

# 実装上の注意

1. **test 5 件は変更しない**: audio.rs / system_audio.rs の既存 5 件 × 2 = 10 件は payload の構造を assert する build_audio_drop_event_payload の test です。event 名とは独立です。touch しないでください。
2. **build_audio_drop_event_payload の use 行は維持**: audio.rs line 587 と system_audio.rs line 453 の `use crate::audio_event::build_audio_drop_event_payload;` は既存テストで使うのでそのままにしてください。
3. **AUDIO_DROP_EVENT_NAME を `use` で import しない**: 呼び出し側 (audio.rs / system_audio.rs) では fully-qualified path `crate::audio_event::AUDIO_DROP_EVENT_NAME` のまま使ってください。`use crate::audio_event::AUDIO_DROP_EVENT_NAME;` を追加すると tests mod 側との use 重複や `unused_imports` の警告経路が増えるリスクがあります。fully-qualified path のほうが、build_audio_drop_event_payload の呼び出し側も同じ fully-qualified スタイルで揃っています。
4. **eprintln マクロは触らない**: line 399 / 329-331 のデバッグログは並存維持です (前ループ判断の累積)。
5. **cfg ガード**: system_audio.rs の emit 呼び出しはおそらく `#[cfg(target_os = "macos")]` の文脈にあります。定数 AUDIO_DROP_EVENT_NAME 自体は platform-agnostic に audio_event.rs に置くだけで OK (cfg ガード不要)。non-macos でも参照されない (system_audio.rs の cfg ガード内のみで使用) ので dead_code 警告も出ない想定。
6. **frontend (TranscriptView.tsx) は touch しない**: 別言語空間。手動同期責任。
7. **コメント追加禁止**: 自明なコメントは不要です。
