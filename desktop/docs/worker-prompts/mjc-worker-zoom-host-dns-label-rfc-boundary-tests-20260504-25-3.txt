# 追加する test 3 件 (具体仕様)

## 背景
- `classify_meeting_url` は Zoom 経路で `is_zoom_host` (l.681) → `is_valid_dns_label` (l.696) を呼ぶ。
- `is_zoom_host` (l.681): `zoom.us` / `zoomgov.com` 完全一致 OR `<sub>.zoom.us` / `<sub>.zoomgov.com` で subdomain の各 label が `is_valid_dns_label` で valid。
- `is_valid_dns_label` (l.696): RFC 1035/1123 の DNS label 規則を実装:
  - `!bytes.is_empty()` (非空)
  - `bytes.len() <= 63` (RFC 1035 上限)
  - 全文字 ASCII alphanumeric or `-`
  - 先頭・末尾は ASCII alphanumeric (`-` 不可)
- 既存 test (l.1305-1335) でカバー済の境界:
  - 空 subdomain `.zoom.us` reject ✅
  - 空 label `evil..zoom.us` reject ✅
  - underscore `bad_label.zoom.us` reject ✅
  - leading hyphen `-bad.zoom.us` reject ✅
  - trailing hyphen `bad-.zoom.us` reject ✅
- 未保護の境界:
  - **長さ最大値 63 バイト ぴったり accept** (RFC 1035 上限ぴったり、`<= 63` の左境界内側、`< 64` の境界)
  - **長さ右境界外側 64 バイト reject** (RFC 1035 上限超過、`<= 63` の `<= 64` 化誤改修を遮断)
  - **数字始まり label accept** (RFC 1123 仕様、RFC 952 では禁止だったがアップデート許容)
- これら 3 つは「対称的補強の 3 軸構造」に綺麗に適用可能 = 「会議検知の信頼性」軸 (優先順位 #2) への直接寄与継続。

## 追加 test 仕様 (mod tests 末尾、Loop 2 last test の直後に追加)

### T1: `classify_meeting_url_accepts_zoom_subdomain_label_at_dns_label_max_length_63_bytes`
- 目的: `is_valid_dns_label` の `bytes.len() <= 63` 上限ぴったり (左境界内側) accept 動作を CI 固定。RFC 1035 仕様の executable spec。
- 仕様: 63 バイトの ASCII 小文字 label (例: `"a".repeat(63)`) を subdomain として持つ Zoom URL `https://<63 文字 a>.zoom.us/j/123456789` を accept する現契約を CI 固定。
- 期待: `Some(MeetingUrlClassification { service: "Zoom".to_string(), host: format!("{}.zoom.us", "a".repeat(63)) })` を返す。
- 実装ヒント:
```rust
let label = "a".repeat(63);
let url = format!("https://{}.zoom.us/j/123456789", label);
let expected_host = format!("{}.zoom.us", label);
assert_eq!(
    classify_meeting_url(&url),
    Some(MeetingUrlClassification {
        service: "Zoom".to_string(),
        host: expected_host,
    }),
    "is_valid_dns_label は RFC 1035 上限 63 バイトぴったり (現契約: bytes.len() <= 63) を accept する必要がある"
);
```

### T2: `classify_meeting_url_rejects_zoom_subdomain_label_exceeding_dns_label_max_length_64_bytes`
- 目的: `is_valid_dns_label` の `bytes.len() <= 63` 上限超過 (右境界外側) reject 動作を CI 固定。「`<= 63` を `<= 64` に変える誤改修」を遮断する装置。
- 仕様: 64 バイトの ASCII 小文字 label (例: `"a".repeat(64)`) を subdomain として持つ Zoom URL `https://<64 文字 a>.zoom.us/j/123456789` を `None` で reject する現契約を CI 固定。
- 期待: `classify_meeting_url(...) == None`。
- 実装ヒント:
```rust
let label = "a".repeat(64);
let url = format!("https://{}.zoom.us/j/123456789", label);
assert_eq!(
    classify_meeting_url(&url),
    None,
    "is_valid_dns_label は RFC 1035 上限 63 バイト超過 (現契約: bytes.len() <= 63) = 64 バイトは reject する必要がある"
);
```

### T3: `classify_meeting_url_accepts_zoom_subdomain_label_starting_with_digit_per_rfc_1123`
- 目的: `is_valid_dns_label` の `bytes.first().is_some_and(|byte| byte.is_ascii_alphanumeric())` の数字始まり許容動作 (RFC 1123 仕様) を CI 固定。「数字始まり reject」厳格化誤改修を遮断。
- 仕様: 数字始まり label (例: `1company`) を subdomain として持つ Zoom URL `https://1company.zoom.us/j/123456789` を accept する現契約を CI 固定。
- 期待: `Some(MeetingUrlClassification { service: "Zoom".to_string(), host: "1company.zoom.us".to_string() })` を返す。
- 実装ヒント:
```rust
assert_eq!(
    classify_meeting_url("https://1company.zoom.us/j/123456789"),
    Some(MeetingUrlClassification {
        service: "Zoom".to_string(),
        host: "1company.zoom.us".to_string(),
    }),
    "is_valid_dns_label は RFC 1123 仕様で数字始まり label を accept する必要がある (現契約: bytes.first().is_some_and(is_ascii_alphanumeric))"
);
```
