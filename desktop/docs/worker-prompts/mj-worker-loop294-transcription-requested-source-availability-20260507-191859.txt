目的:
`start_transcription` で UI/呼び出し側が要求した transcription source と、実際に利用可能な audio capture がずれた場合に、片側だけで黙って開始しないようにする。特に `source: "both"` なのに mic または system_audio の sample rate が欠けた場合、別トラック文字起こしが片側だけ開始される可能性を避け、明示的なエラーにする。

現状:
- `parse_requested_transcription_sources` は `microphone` / `system_audio` / `both` を bool へ変換する。
- `start_transcription` は requested source ごとに `audio_state.get_sample_rate()` / `get_system_audio_sample_rate()` を読む。
- その後、存在する sample rate だけで stream を作るため、`both` 要求でも片側 missing なら available stream だけで進み得る。
- 最後の `workers.is_empty()` は「全部 missing」だけを検出する。

実装方針:
1. `transcription_commands_helpers.rs` に、要求 source と available source の整合性を検証する純粋関数を追加する。
   - 例: `validate_requested_sources_available(requested: RequestedTranscriptionSources, mic_available: bool, system_available: bool) -> Result<(), String>`
   - `requested.use_mic && !mic_available` ならエラー。
   - `requested.use_system && !system_available` ならエラー。
   - 両方 missing なら既存の「音声ソースが利用可能ではありません。録音を先に開始してください。」に近い文言でもよいが、片側 missing の場合はどちらが欠けたか分かる文言にする。
2. `start_transcription` で sample rate 取得直後、`validate_stream_count_for_engine` より前にその helper を呼ぶ。
3. 既存の `workers.is_empty()` は defensive guard として残してよい。
4. Apple Speech の dual-source reject 文言や `parse_requested_transcription_sources` の契約は変えない。
5. UI、TypeScript、audio capture、worker loop、engine 実装、依存関係は変更しない。

テスト:

- 少なくとも以下を固定:
  - mic only requested + mic available -> Ok
  - system only requested + system available -> Ok
  - both requested + both available -> Ok
  - mic only requested + mic missing -> Err に「マイク」または `microphone` が含まれる
  - system only requested + system missing -> Err に「相手側」または `system_audio` が含まれる
  - both requested + mic missing -> Err
  - both requested + system missing -> Err
  - both requested + both missing -> Err
