## 1. 背景 / Why
`Session` 構造体 (line 13-20) には `id`, `title`, `started_at`, `ended_at`, `segments` の 5 フィールドがあり、3 つの mutating method (`start` 関連は constructor、`append_segment`, `finalize`) が動く。

既存 8 件のテストは個別 method の単発挙動はカバーしているが、**cross-fn invariant** (= ある method を呼んでも他フィールドが変わらない) の保護が薄い。

例えば:
- `append_segment` は `segments` のみを変更し、`started_at` / `id` / `title` を変えない
- `finalize` は `ended_at` のみを変更し、`segments` / `started_at` を変えない
- `start` の title はバリデーションせず passthrough する (空文字許容)
- `append_segment` の speaker / text もバリデーションせず passthrough (空文字許容)
- `append_segment` の `timestamp_offset_secs` は u64 全範囲を許容 (boundary 0 / u64::MAX)

これらの不変条件・上位層責任境界を 5 件の direct test で固定する。

**SESSION_COUNTER 並列実行注意点 (重要)**: `static SESSION_COUNTER: AtomicU64 = AtomicU64::new(0)` (line 4) はテスト間で共有されるグローバルカウンタ。
- **`session.id` の絶対値 (例: "1000-3") を assert してはいけない** (並列実行で seq が不定)。
- 「`id != id2`」「`id` が `started_at-` で始まる」のような **相対 / prefix assert は OK**。
- 既存 line 110-118 (`start_assigns_unique_ids_across_consecutive_calls`) と line 121-134 (`start_id_starts_with_started_at_prefix_and_hyphen`) と同じ流儀を踏襲する。

## 2. 具体的 test 案 (assertion レベル)

`super::*` は既に use 済み。

### T1. `append_segment_preserves_started_at_id_and_title_invariants`
```rust
let mut session = Session::start("会議メモ".into(), 1_000);
let original_id = session.id.clone();
let original_started_at = session.started_at;
let original_title = session.title.clone();

for i in 0..5 {
    session.append_segment(format!("speaker{}", i), i, format!("text{}", i));
}

assert_eq!(session.id, original_id, "append_segment は id を変えない");
assert_eq!(session.started_at, original_started_at, "append_segment は started_at を変えない");
assert_eq!(session.title, original_title, "append_segment は title を変えない");
assert_eq!(session.ended_at, None, "append_segment は ended_at に触らない");
assert_eq!(session.segments.len(), 5, "append された segment 数は維持");
```
**意図**: `append_segment` の単一責務 (segments のみ変更) を 5 フィールド全て確認する cross-fn invariant 契約。将来 `append_segment` が「最初の segment 時に title を auto-update」のような副作用を持ち込もうとしたらこの test で検知。

### T2. `finalize_preserves_started_at_and_segments_invariants`
```rust
let mut session = Session::start("title".into(), 1_000);
session.append_segment("Alice".into(), 5, "hello".into());
session.append_segment("Bob".into(), 12, "world".into());

let original_id = session.id.clone();
let original_started_at = session.started_at;
let original_segments_len = session.segments.len();
let original_first_text = session.segments[0].text.clone();

session.finalize(2_000);

assert_eq!(session.id, original_id, "finalize は id を変えない");
assert_eq!(session.started_at, original_started_at, "finalize は started_at を変えない");
assert_eq!(session.segments.len(), original_segments_len, "finalize は segments の長さを変えない");
assert_eq!(session.segments[0].text, original_first_text, "finalize は既存 segment 内容を変えない");
assert_eq!(session.ended_at, Some(2_000), "finalize は ended_at だけを設定する");
```
**意図**: `finalize` の単一責務 (ended_at のみ変更) を確認する cross-fn invariant 契約。将来 `finalize` が「segments を sort」「title を normalize」のような副作用を持ち込もうとしたらこの test で検知。

### T3. `start_accepts_empty_title_as_passthrough`
```rust
let session = Session::start(String::new(), 1_000);
assert_eq!(session.title, "", "Session::start は title バリデーションせず空文字を passthrough");
assert_eq!(session.started_at, 1_000);
assert!(!session.id.is_empty(), "title が空でも id は生成される");
assert!(session.segments.is_empty());
assert_eq!(session.ended_at, None);
```
**意図**: title の validation/normalization は上位層責任である現契約を固定 (前セッション「上位層責任境界の契約強制」パターン再利用)。将来 `Session::start` が「空 title なら "Untitled" にフォールバック」のような責務を持ち込もうとしたらこの test で検知。

### T4. `append_segment_accepts_empty_speaker_and_text_as_passthrough`
```rust
let mut session = Session::start("title".into(), 1_000);
session.append_segment(String::new(), 5, String::new());

assert_eq!(session.segments.len(), 1, "空文字 speaker/text でも segment は push される");
assert_eq!(session.segments[0].speaker, "", "speaker の空文字は passthrough");
assert_eq!(session.segments[0].text, "", "text の空文字は passthrough");
assert_eq!(session.segments[0].timestamp_offset_secs, 5);
```
**意図**: `append_segment` の speaker/text 引数は validation せず passthrough する現契約を固定。speaker fallback ("不明" など) は `transcript_bridge.rs::normalize_speaker` の責務 (上位層) であり、`Session::append_segment` 自身は責務を持たない境界を明示。

### T5. `append_segment_accepts_zero_and_max_u64_offset_as_passthrough`
```rust
let mut session = Session::start("title".into(), 1_000);
session.append_segment("a".into(), 0, "zero".into());
session.append_segment("b".into(), u64::MAX, "max".into());

assert_eq!(session.segments.len(), 2);
assert_eq!(session.segments[0].timestamp_offset_secs, 0, "offset=0 boundary を passthrough");
assert_eq!(session.segments[1].timestamp_offset_secs, u64::MAX, "offset=u64::MAX boundary を passthrough");
```
**意図**: `timestamp_offset_secs` の値域は u64 全範囲という現契約を boundary 値 (0 と u64::MAX) で固定。将来「offset > N 秒は reject」のような validation を入れたらこの test で検知。

## 3. 実装上の注意

- `super::*` は line 60 で既に use 済み。追加 import は不要。
- `for i in 0..5` の `i` は `u64` 型を要求 (`session.append_segment(format!("..."), i, ...)`)。Rust の型推論で `0..5` は `Range<i32>` がデフォルトなので、`for i in 0u64..5u64` または `i as u64` への明示変換が必要。最も読みやすいのは `for i in 0u64..5` (右辺は型推論で u64)。
- T1 の `format!("speaker{}", i)` は clippy::uninlined_format_args が走る環境でも問題なし (引数 1 個の inline format でも prefix が変数なら OK)。安全のため `format!("speaker{i}")` の inline 形式を使うこと。
- **コメントは日本語、簡潔に** (AGENTS.md 準拠)。1 関数あたり 1-3 行のコメントで十分。
- **SESSION_COUNTER 絶対値を assert しない** (上記注意点参照)。`session.id` の絶対値比較は禁止。`assert_ne!` や `starts_with` は OK。
- **既存テストとの重複回避**:
  - T2 の「finalize 後 segments 維持」は line 137-153 の `append_segment_after_finalize_records_post_finalize_segment` と似ているが、後者は「finalize 後に append が動く」を確認。T2 は「finalize が既存 segments を変えない」を確認 (観点が逆)。
  - T3 と既存 line 63-70 の `start_creates_session_with_initial_state` は重なるが、後者は title="title" で空文字 boundary を確認していない。T3 は空文字 boundary 専用。
