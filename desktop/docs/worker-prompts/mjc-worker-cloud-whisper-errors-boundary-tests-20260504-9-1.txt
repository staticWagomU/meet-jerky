# 1. 背景 / Why

`sanitize_error_body` (line 25-37) は `cloud_whisper_errors.rs` の core helper で、HTTP error body を一定長に切り詰めるサニタイズロジックを担う。`MAX_ERROR_BODY_CHARS = 200` (line 23) を境に truncation するが、現契約は以下の非自明な振る舞いを持つ:

1. **truncation 条件は `>` であり `>=` ではない (line 27)**: 200 chars ジャストでは truncation されない。
2. **whitespace 正規化が char count 判定より前に走る (line 26 → 27)**: 入力に余分 whitespace を含めても normalized 後の char count で判定する。
3. **multibyte (CJK 等) は char count であり byte count ではない**。
4. **truncate された場合は `{sanitized}...` の形式 = output 長 = MAX + 3 chars (= 203)**。

既存テスト (8 件) は以下を確認済:
- `classify_other_error_sanitizes_body`: `MAX + 20` (220 chars) で truncation 確認、output が `MAX + 3` chars であること、"..." で終わることを assert。
- `sanitize_error_body_truncates_multibyte_text_by_char_count_not_byte_count`: "あ" × `MAX + 5` (= 205 chars) で multibyte truncation。
- `sanitize_error_body_falls_back_to_empty_marker_for_whitespace_only_inputs`: 空 / spaces / tab+newline+spaces。
- `classify_other_error_sanitizes_body` 後段: whitespace 正規化 ("first line\n\tsecond line  " → "first line second line")。

**確認した gap**:
- **199 / 200 / 201 chars のピンポイント boundary** が直接テストされていない。特に「200 chars ジャストでは truncate されない」という非対称契約 (`>` vs `>=`) は将来のリファクタで誤って `>=` に変更されると静かに振る舞いが変わる。
- **whitespace 正規化が char count 判定より先に走る順序契約** が直接テストされていない (raw byte/char で MAX + α だが正規化後に MAX 以下になるケース)。
- **multibyte の boundary 上ジャスト (201 chars)** が `+ 5` で間接確認のみ。

これら 5 件で boundary を `>=` 誤改修・順序入れ替え誤改修・multibyte 誤改修すべて検知できる装置を作る。

# 2. 追加する 5 件のテスト案 (assertion レベル)

すべて `cloud_whisper_errors.rs` の `mod tests` 末尾に追加。private fn `sanitize_error_body` と private const `MAX_ERROR_BODY_CHARS` は同じファイル内の `tests` mod から `super::` で直接呼べる (既存 `super::sanitize_error_body` / `super::MAX_ERROR_BODY_CHARS` のパターン踏襲)。

## T1. `sanitize_error_body_does_not_truncate_at_199_chars`
- 入力: `"x".repeat(MAX_ERROR_BODY_CHARS - 1)` (= 199 chars)
- assert:
  - `result.chars().count() == 199`
  - `result == "x".repeat(199)` (内容そのまま、`...` 含まれない)
  - `!result.ends_with("...")` (truncation マーカーなし)
- 意図: 境界一歩下 (199) で truncation が起きない現契約を固定。

## T2. `sanitize_error_body_does_not_truncate_at_exactly_200_chars`
- 入力: `"x".repeat(MAX_ERROR_BODY_CHARS)` (= 200 chars ジャスト)
- assert:
  - `result.chars().count() == 200`
  - `result == "x".repeat(200)` (内容そのまま)
  - `!result.ends_with("...")` (truncation されない、これが `>` vs `>=` の決定的差)
- 意図: 200 chars ジャストの境界上で truncation 「されない」現契約 (`>` 比較) を固定。これが `>=` 誤改修検知装置の本体。

## T3. `sanitize_error_body_truncates_at_201_chars`
- 入力: `"x".repeat(MAX_ERROR_BODY_CHARS + 1)` (= 201 chars)
- assert:
  - `result.chars().count() == MAX_ERROR_BODY_CHARS + 3` (= 203 chars: 200 + "...")
  - `result.ends_with("...")`
  - `result.starts_with(&"x".repeat(MAX_ERROR_BODY_CHARS))` (最初の 200 chars が x で、その後に "...")
- 意図: 境界一歩上 (201) で truncation が「初めて」発生する現契約を固定。T2 とペアで `>` 境界の上下を完全網羅。

## T4. `sanitize_error_body_normalizes_whitespace_before_counting_chars`
- 入力: 例えば `"x ".repeat(150)` (= 300 chars: x と space の交互。raw char count = 300、whitespace 正規化後は `split_whitespace` により 150 個の "x" が " " で join されて 150 + 149 = 299 chars... ではダメ。`split_whitespace` は連続/末尾 whitespace を除去して token を作るので、`"x ".repeat(150)` = `"x x x ... x "` (末尾 space 込みで 300 chars) → tokens = 150 個の "x" → join " " → 150 + 149 = 299 chars。これは MAX = 200 を超えて truncate される)。
  - 正規化後 199 chars にしたいので、`"x".to_string()` を 100 回 + `" ".to_string()` を 99 回挟む構成だと → tokens 100 個で join 後 100 + 99 = 199 chars。これだと truncate されない。
  - **採用**: 入力 = `"x x x ... x"` (100 個の "x" を 1 space 区切りで並べる) を `let body = (0..100).map(|_| "x").collect::<Vec<_>>().join("    ");` (4 spaces で区切る) → raw char count = 100 + 99*4 = 496 chars (確かに MAX = 200 を超える)、しかし `split_whitespace().join(" ")` 後は 100 個の "x" を 1 space で繋いだ 199 chars (`= 100 + 99`) → truncate されない。
- assert:
  - 入力 raw `body.chars().count() == 496` (raw は MAX 超え)
  - `result.chars().count() == 199` (正規化後は MAX 未満)
  - `!result.ends_with("...")` (truncation されない、これが「正規化が char count 判定より先に走る」順序契約の決定的差)
  - `result == (0..100).map(|_| "x").collect::<Vec<_>>().join(" ")` (内容も期待通り)
- 意図: line 26 (whitespace 正規化) → line 27 (char count 判定) の順序契約を固定。順序入れ替え (raw chars で先に判定 → 後で正規化) 誤改修を検知。

## T5. `sanitize_error_body_truncates_multibyte_at_exactly_201_chars`
- 入力: `"あ".repeat(MAX_ERROR_BODY_CHARS + 1)` (= 201 chars)
- assert:
  - `result.chars().count() == MAX_ERROR_BODY_CHARS + 3` (= 203 chars: 200 個の "あ" + "...")
  - `result.ends_with("...")`
  - `result.starts_with("あ")` (multibyte の先頭は維持)
  - `result.chars().take(MAX_ERROR_BODY_CHARS).all(|c| c == 'あ')` (200 個目までは すべて "あ")
- 意図: 既存 multibyte test は `+ 5` で truncation 動作のみ確認しているが、boundary ジャスト上 (`+ 1`) で truncation が初めて発生する現契約を multibyte 専用で固定。byte count 誤改修 (e.g. UTF-8 の 1 文字 = 3 bytes として扱う) を検知。

- 既存 8 件テストとの重複を避けるため、200 chars 未満入力 / 200 chars ジャスト / 201 chars という「分岐の境目」だけに絞る。
