# 背景 / Why

システム音声 (デスクトップ音声) はプロダクト優先度 3 (マイク音声とデスクトップ音声の別トラック取得) の
直結機能です。本ループでは 2 つの未カバー領域を直接テストで補完します。

## 1. `fn read_f32_ne(sample_bytes: &[u8]) -> f32` (L35-L42)

```rust
fn read_f32_ne(sample_bytes: &[u8]) -> f32 {
    f32::from_ne_bytes([
        sample_bytes[0], sample_bytes[1], sample_bytes[2], sample_bytes[3],
    ])
}
```

現状は `f32_pcm_bytes_to_mono` 経由の間接テストのみ (8 件)。ただし `f32_pcm_bytes_to_mono` は
読み取り後 `sanitize_pcm_sample` で NaN/Inf を 0.0 に潰すため、**read_f32_ne 段階の
bit-pattern 読み取り正確性は間接テストでは保証されない** 。

直接テストの価値:
- bit-pattern 仕様の固定 (0.0 / 1.0 / NaN / -Inf を round-trip で確認)
- sanitize 前段の正確性保護 (`from_ne_bytes` を `from_le_bytes` に置き換える誤リファクタを検知)

## 2. `fn validate_audio_format_properties(...)` (L80-L104)

```rust
if !is_float { return Err("非 f32 PCM..."); }     // ① 第 1 関門
if is_big_endian { return Err("BigEndian..."); }   // ② 第 2 関門
match bits_per_channel { ... }                     // ③ 第 3 関門
match channel_count { ... }                        // ④ 第 4 関門
Ok(())
```

既存 8 件は **個別 violation** の test ですが、**複数 violation 同時時の short-circuit 順序**
(関門 ① が ② より先に発火する契約) は未テスト。順序入れ替えリファクタを検知できない。

# 具体的な実装変更

実装は変更しません。**`mod tests` 末尾**に以下の 5 件を追加してください。

各テストは独立した `#[test] fn` とすること。`use super::*;` は既存で対応済み (test mod 先頭で確認)。

### T1. read_f32_ne_decodes_zero_bytes_to_zero_float

```rust
#[test]
fn read_f32_ne_decodes_zero_bytes_to_zero_float() {
    // 4 byte 全 0 は f32 では 0.0 (どの endian でも符号なし)。
    // bit-pattern 読み取りの基本契約を固定する。
    let bytes = [0u8; 4];
    assert_eq!(read_f32_ne(&bytes), 0.0);
}
```

### T2. read_f32_ne_round_trips_one_point_zero

```rust
#[test]
fn read_f32_ne_round_trips_one_point_zero() {
    // f32::to_ne_bytes と read_f32_ne (= f32::from_ne_bytes) の双方向 round-trip を確認。
    // from_le_bytes / from_be_bytes への誤リファクタを native endian 環境で検知する
    // (multibyte cross-platform でも to_ne_bytes ↔ from_ne_bytes は常に round-trip 不変)。
    let original: f32 = 1.0;
    let bytes = original.to_ne_bytes();
    assert_eq!(read_f32_ne(&bytes), original);
}
```

### T3. read_f32_ne_decodes_nan_bit_pattern_to_nan

```rust
#[test]
fn read_f32_ne_decodes_nan_bit_pattern_to_nan() {
    // NaN の bit-pattern は f32::NAN.to_ne_bytes() で取得し、read 結果が is_nan() を満たす。
    // sanitize_pcm_sample の前段で NaN bit-pattern が正しく NaN として読まれることが
    // 間接テストでは保証されない (sanitize で 0.0 に潰されてしまうため)。直接テストで保護。
    let bytes = f32::NAN.to_ne_bytes();
    assert!(read_f32_ne(&bytes).is_nan());
}
```

### T4. read_f32_ne_decodes_negative_infinity_bit_pattern

```rust
#[test]
fn read_f32_ne_decodes_negative_infinity_bit_pattern() {
    // -Inf の bit-pattern が is_infinite() && is_sign_negative() を満たす。
    // f32 IEEE 754 表現の符号ビット保持を確認 (sanitize 前段の保証)。
    let bytes = f32::NEG_INFINITY.to_ne_bytes();
    let result = read_f32_ne(&bytes);
    assert!(result.is_infinite());
    assert!(result.is_sign_negative());
}
```

### T5. validate_audio_format_properties_returns_first_violation_when_multiple_invalid

```rust
#[test]
fn validate_audio_format_properties_returns_first_violation_when_multiple_invalid() {
    // 4 関門 (is_float / is_big_endian / bits_per_channel / channel_count) すべて violation の場合、
    // 第 1 関門 (is_float check) が最初に発火し "非 f32 PCM..." を返す現挙動を固定。
    // 関門順序の入れ替えリファクタを CI で検知する装置として機能。
    let result = validate_audio_format_properties(
        /* is_float */ false,
        /* is_big_endian */ true,
        /* bits_per_channel */ Some(16),
        /* channel_count */ Some(99),
        /* expected_channels */ 2,
    );

    // 第 1 関門が最優先で発火することを文言完全一致で固定。
    assert_eq!(
        result,
        Err("非 f32 PCM フォーマット (kAudioFormatFlagIsFloat 未設定)")
    );
}
```

# 実装上の注意

- **配置場所**: 既存 `mod tests` の最後の `}` の直前。最後のテストの直後。
- `use super::*;` は既存 mod tests 先頭で対応済 (確認すること)。新規 use 不要。
- `f32::NAN`, `f32::NEG_INFINITY`, `f32::to_ne_bytes()` は std で標準利用可能。
- 実装 (`fn read_f32_ne`, `fn validate_audio_format_properties`) は変更しない。
- 他のテスト関数を変更しない。
- T5 のエラー文言は既存の test (`test_validate_audio_format_rejects_non_float` など) で
  使われているはず。同じ文言を使うこと (実装の `&'static str` リテラルと一致する完全一致)。
- `read_f32_ne` の visibility は private のまま OK (`use super::*;` 経由で同 mod 内テスト可能)。
- T2 の round-trip は cross-platform で常に成立するので、CI が大小エンディアン両方で動いても
  問題ない。
