# 背景

直前の Loop 2 (commit 4482bb0) で F-Loop3 (会議終了検知のアプリ統合) の機能完結が完了しました:
- `start()` で std::thread タイマーを 60 秒周期で起動
- `check_all_inactive_bundles()` が WATCHED_BUNDLE_IDS (Zoom / Teams2 / Teams legacy / FaceTime) を巡回
- 各 bundle に対して `check_meeting_inactive_for_bundle` 経由で純粋関数 `should_notify_meeting_inactive` を呼ぶ
- Some(elapsed) なら `show_inactive_notification` 発火 (10 分 = MEETING_INACTIVE_THRESHOLD)

ただし `handle_browser_url_detection` (browser url 経路) では `last_seen_secs` を **更新していません**。これは前 mjc-main-20260504-19 Loop 2 (commit 09d191c, handle_detection 側 update) の **対称コピーが未実装** な状態。

結果、ブラウザで会議参加中 (例: Safari で Google Meet) のセッションは inactive 通知が発火しません (= 会議が終了してもユーザーが録音停止を忘れる体験リスク)。

# Why (この変更の動機)

- F-Loop3 機能の対称性回復: `handle_detection` (アプリ経路) と `handle_browser_url_detection` (browser 経路) の **両方が同じ runtime invariant を満たす** 状態にする = 設計の整合性。
- Tidy First 厳守: 本 Loop ではデータ更新のみ。`check_all_inactive_bundles` 側の iteration extension (key 全件巡回 + display name 抽出) は **本 Loop scope 外** = 将来の別 Loop に分離。
- 価値の段階的展開: 本 Loop でデータが収集されるようになる → 将来 iteration extension で初めて inactive 通知発火がブラウザ経路でも動く。前 mjc-main-19 Loop 2 と同じ「Loop 内ではユーザー観測不可、後続 Loop で観測可能化」パターン。

## 1. `handle_browser_url_detection` 内に `last_seen_secs.insert(throttle_key, now_secs)` 追加

`handle_browser_url_detection` 関数 (line 225 付近〜317 付近) の中で:

- 既存の `last_seen.insert(throttle_key, now)` (line 298 付近、`last_seen` MutexGuard を使う `Instant` ベースの insert) **の閉じ括弧 `}` の直後** に、新規ブロックを追加して `last_seen_secs` を更新する。
- パターンは **mjc-main-19 Loop 2 (commit 09d191c) の handle_detection 内 update と完全対称**:

```rust
// `should_notify_meeting_inactive` 用に epoch secs ベースの最終検知時刻を更新する
// (案 A 二重管理: 既存 Instant ベース throttle と並走)。
{
    let now_secs = std::time::SystemTime::now()
        .duration_since(std::time::UNIX_EPOCH)
        .map(|d| d.as_secs())
        .unwrap_or(0);
    state
        .last_seen_secs
        .lock()
        .insert(throttle_key.clone(), now_secs);
}
```

**重要**:
- key には **`throttle_key` を使う** (= `"browser:<bundle_id>:<service>:<host>"` または `"window-title:<bundle_id>:<service>"` 形式)。`bundle_id` 単独ではなく throttle_key を使う理由は、ブラウザ内の複数会議を区別するため + 既存 `last_seen` HashMap の key と完全一致するため = 二重管理の対称性。
- `throttle_key` は line 274-283 で `format!` で生成され、line 298 で `last_seen.insert(throttle_key, now)` で move 消費される可能性あり = **`.clone()` を取って使う** (上記コード例の通り)。あるいは line 298 を `last_seen.insert(throttle_key.clone(), now)` に変更してから後続で参照する形にする。**どちらでも OK**、cargo clippy が通る方を選ぶ。
- 既存 `let mut last_seen = state.last_seen.lock();` ブロック (line 290-298 付近) を抜けた **後** に新規 `last_seen_secs.lock()` を取る = lock 順序を保つ (last_seen の MutexGuard が drop してから last_seen_secs を取得 = デッドロック回避)。
- `let state = match STATE.get() { ... }` (line 285-288 付近) は既に取得済 = 同じ state 変数を再利用。

# 実装上の注意

- **Tidy First 厳守**: ユーザー観測可能な振る舞い不変。誰も browser 経路の last_seen_secs エントリを読まない (`check_all_inactive_bundles` が WATCHED_BUNDLE_IDS のみ巡回するため)。
- **既存パターン踏襲**: mjc-main-19 Loop 2 (commit 09d191c) で handle_detection 内に追加した update ブロックと **完全対称な構造**:
  - `let now_secs = std::time::SystemTime::now().duration_since(std::time::UNIX_EPOCH).map(|d| d.as_secs()).unwrap_or(0);`
  - `state.last_seen_secs.lock().insert(<key>, now_secs);`
  - 別スコープ `{ ... }` で囲んで lock を即解放。
- **lock 順序維持**: 既存 `last_seen.lock()` (Instant 系) → `latest_payload.lock()` (line 310 付近) の順序に新規 `last_seen_secs.lock()` を **干渉させない**。`last_seen` MutexGuard が drop してから取得すること。
- **throttle_key の clone**: `format!` で生成された String は move 消費される。clone を取るか、insert 順序を入れ替える必要あり。
- **担当範囲外を編集しない** (lib.rs / 他のファイル / Swift bridge / `check_all_inactive_bundles` の iteration ロジックは触らない)。
- **コミット禁止**。実装と検証だけ。
