背景 / Why this matters:
- meet-jerky はクラウド文字起こし API (OpenAI Whisper / ElevenLabs) のレスポンスを `classify_cloud_whisper_error` で 4 variant (`InvalidApiKey` / `RateLimited` / `ServerError` / `Other`) に分類する。これは UI のエラー表示・課金 (rate limit) の信頼性に直結する関数。
- 既存 4 件のテストで 401, 429, 500, 599, 400, 418 と sanitize_error_body の経由パスはカバー済み。
- 未カバーの致命的 gap が以下:
  1. `5xx` 範囲の隣接境界 (`499`, `600`) — 範囲リファクタへの検知装置なし。
  2. private fn `sanitize_error_body` の直接呼び出しテストなし — 関数抽出/インライン化リファクタの責任分離が崩れた時に切り分け不能。
  3. multibyte 文字列の切り詰めが char base である契約 (`chars().take(MAX_ERROR_BODY_CHARS)`) を test で固定する装置なし — byte base への誤リファクタで invalid UTF-8 panic の致命バグに至る。
  4. 200 系 success status を error path に誤呼出された場合に Other variant で safe に分類される契約が未確認。

追加してほしいテスト (5 件、~10-15 分で完了):

1. `classify_499_returns_other` (5xx 直前境界):
   - `assert_eq!(classify_cloud_whisper_error(499, "boundary-before-5xx"), CloudWhisperError::Other { status: 499, message: "boundary-before-5xx".to_string() })`
   - Why: 5xx 範囲の lower bound (500) と Other (499) の境界が壊れた瞬間に検知できる。

2. `classify_600_returns_other` (5xx 直後境界):
   - `assert_eq!(classify_cloud_whisper_error(600, "boundary-after-5xx"), CloudWhisperError::Other { status: 600, message: "boundary-after-5xx".to_string() })`
   - Why: 5xx 範囲の upper bound (599) を 600 や 999 まで広げる誤リファクタを即座に検知。

3. `classify_200_returns_other_safely` (success status を誤呼出された場合の安全網):
   - `assert_eq!(classify_cloud_whisper_error(200, "unexpected success in error path"), CloudWhisperError::Other { status: 200, message: "unexpected success in error path".to_string() })`
   - Why: 上位層 (transcription pipeline) で誤って success status を error path に渡しても crash せず Other に分類される契約。実装上 `match` の `_` 分岐で吸収されるが、明示的に test で固定すると将来 `_` を `400..=499` に絞る誤リファクタを検知できる。

4. `sanitize_error_body_falls_back_to_empty_marker_for_whitespace_only_inputs` (private fn 直接テスト):
   - `assert_eq!(super::sanitize_error_body(""), "HTTP error body was empty")`
   - `assert_eq!(super::sanitize_error_body("   "), "HTTP error body was empty")`
   - `assert_eq!(super::sanitize_error_body("\t\n  "), "HTTP error body was empty")`
   - Why: 既存テストでは `classify_cloud_whisper_error(418, "")` 経由でしか empty marker フォールバックが確認されていない。private fn 直接呼び出しで `split_whitespace().collect::<Vec<_>>().join(" ")` の whitespace 多様性 (タブ/改行混在) を網羅し、関数抽出リファクタへの耐性を獲得する。

5. `sanitize_error_body_truncates_multibyte_text_by_char_count_not_byte_count` (multibyte 切り詰めの致命的安全網):
   - `let body = "あ".repeat(super::MAX_ERROR_BODY_CHARS + 5);` (205 chars × 3 bytes/char = 615 bytes)
   - `let result = super::sanitize_error_body(&body);`
   - `assert_eq!(result.chars().count(), super::MAX_ERROR_BODY_CHARS + 3);` (200 個の "あ" + "...")
   - `assert!(result.ends_with("..."));`
   - `assert!(result.starts_with("あ"));` (1 文字目が完全な "あ" であること = byte 切りでないこと)
   - Why: 現実装は `chars().take(MAX_ERROR_BODY_CHARS)` で char base 切り詰め。これを `.bytes().take(N)` などに誤リファクタすると invalid UTF-8 で panic する致命バグになる。3 バイト文字 "あ" を 200 個並べることで、byte base 切り詰めへの誤リファクタを最も簡潔に検知できる。

実装上の注意:
- `MAX_ERROR_BODY_CHARS` は private const だが、`#[cfg(test)] mod tests` 内では `super::MAX_ERROR_BODY_CHARS` でアクセス可能 (既存 test L95 で使用済)。
