## 追加するテスト 3 件 (関数名はそのまま使う)

### A. `set_api_keys_return_consistent_japanese_message_for_empty_input`
- 振る舞い: `set_openai_api_key("")` と `set_elevenlabs_api_key("")` の両方が `Err` を返し、その文言が **完全一致で `"API キーが空です"`** であること
- 不変条件:
  - `set_openai_api_key(String::new()).unwrap_err() == "API キーが空です"`
  - `set_elevenlabs_api_key(String::new()).unwrap_err() == "API キーが空です"`
  - 両 API の文言が同じであること (UI 上の体験統一性)
- 設計意図: 既存 test は `is_err()` のみ。UI 表示文言が壊れても気付けない。文言完全一致の契約強制で UI 体験を保護。

### B. `secret_key_accounts_are_distinct`
- 振る舞い: `SecretKey::OpenAIApiKey.account() != SecretKey::ElevenLabsApiKey.account()`
- 不変条件: 2 つの SecretKey variant が同じ account 文字列を返さない (copy-paste で衝突した場合の検知)
- 設計意図: 既存 `secret_key_account_is_stable` は両者の絶対値を個別固定するが、「両者が異なる」不変条件を明示していない。例えば `OpenAIApiKey.account()` のリテラルを `"elevenlabs-api-key"` に書き換えると、`secret_key_account_is_stable` だけが落ちて `accounts_are_distinct` は両方落ちる → エラー切り分けが容易。

### C. `set_api_keys_treat_various_whitespace_chars_as_empty`
- 振る舞い: 以下 8 assertion (4 種類の whitespace × 2 API)
  - `set_openai_api_key("\t".to_string()).is_err()` (タブのみ)
  - `set_openai_api_key("\n".to_string()).is_err()` (改行のみ)
  - `set_openai_api_key("\t\n  ".to_string()).is_err()` (混在)
  - `set_openai_api_key("  \t \n ".to_string()).is_err()` (混在 + spaces)
  - `set_elevenlabs_api_key("\t".to_string()).is_err()`
  - `set_elevenlabs_api_key("\n".to_string()).is_err()`
  - `set_elevenlabs_api_key("\t\n  ".to_string()).is_err()`
  - `set_elevenlabs_api_key("  \t \n ".to_string()).is_err()`
- 設計意図: 既存 test は `"   "` (半角空白のみ) しか確認していない。`trim()` の仕様変更 (例: ASCII whitespace のみから半角空白のみに狭める) を検知できない。tab/newline を含む多様な whitespace で `trim().is_empty()` の挙動を裏付ける。
