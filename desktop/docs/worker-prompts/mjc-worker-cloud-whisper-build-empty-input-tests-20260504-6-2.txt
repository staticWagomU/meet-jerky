# 背景 / Why

`build_whisper_api_url(base_url: &str) -> String` (L18-L21) と
`build_whisper_authorization_header(api_key: &str) -> String` (L24-L26) は
現状 21 件の test を持つ核心 fn ですが、**空入力 / 異常入力 / 多重 trailing slash
の現挙動が未カバー**です。

これらの fn は内部で validation/trim を一切しない単純な文字列構築のみ:
```rust
pub fn build_whisper_api_url(base_url: &str) -> String {
    let trimmed = base_url.trim_end_matches('/');
    format!("{trimmed}/audio/transcriptions")
}

pub fn build_whisper_authorization_header(api_key: &str) -> String {
    format!("Bearer {api_key}")
}
```

これは「URL/api_key バリデーションは上位層 (呼び出し側) の責任」という設計を意味します。

直接テストを追加する価値:
1. **責任境界の明示**: `""` / `"/"` / 前後 whitespace 込み入力の現挙動を test で固定することで、
   「ここでは validation しない」契約を明示。リファクタで「ここでも空チェックする」変更が
   入った瞬間に test が落ちて気付ける。
2. **trim_end_matches の連続 char 挙動**: `"https://x/v1///"` のように連続スラッシュの
   全削除挙動 (`trim_end_matches('/')` は連続 char をすべて剥がす Rust 標準) を契約として固定。
3. **multibyte 安全性**: `"日本語キー"` のような multibyte api_key が UTF-8 panic を
   起こさず Bearer の後ろに埋め込まれる現挙動の保護。

# 具体的な実装変更

実装は変更しません。**`mod tests` 末尾**に以下の 5 件を追加してください。

各テストは独立した `#[test] fn` とすること。`use super::*;` は既存 (L124) で対応済み。

### T1. build_whisper_api_url_with_empty_base_url_returns_absolute_path

**目的**: 空 base_url で URL validation 無しの現挙動を契約強制。

```rust
#[test]
fn build_whisper_api_url_with_empty_base_url_returns_absolute_path() {
    // base_url = "" でも fn は何も検証せず "/audio/transcriptions" を返す。
    // 上位層で URL バリデーションする責任境界を明示。
    let url = build_whisper_api_url("");

    assert_eq!(url, "/audio/transcriptions");
}
```

### T2. build_whisper_api_url_with_only_slash_returns_absolute_path

**目的**: `"/"` のみの base_url が trim_end_matches で完全に剥がされる現挙動の固定。

```rust
#[test]
fn build_whisper_api_url_with_only_slash_returns_absolute_path() {
    // "/" は trim_end_matches('/') で完全に剥がされ、
    // 結果は "/audio/transcriptions" になる。root slash 1 文字も最後の slash として
    // 消費される現挙動を契約として固定。
    let url = build_whisper_api_url("/");

    assert_eq!(url, "/audio/transcriptions");
}
```

### T3. build_whisper_api_url_strips_multiple_trailing_slashes

**目的**: `trim_end_matches('/')` の連続 char 削除挙動を契約強制。

```rust
#[test]
fn build_whisper_api_url_strips_multiple_trailing_slashes() {
    // trim_end_matches('/') は連続する '/' をすべて削除する Rust 標準ライブラリ挙動。
    // 1 個だけ削除する .strip_suffix("/") への誤リファクタを CI で即座検知。
    let url = build_whisper_api_url("https://api.openai.com/v1///");

    assert_eq!(url, "https://api.openai.com/v1/audio/transcriptions");
}
```

### T4. build_whisper_authorization_header_with_empty_api_key_yields_bearer_with_trailing_space

**目的**: 空 api_key で `"Bearer "` (末尾スペース有) の現挙動を契約強制。

```rust
#[test]
fn build_whisper_authorization_header_with_empty_api_key_yields_bearer_with_trailing_space() {
    // 空 api_key でも fn は validation 無しで `"Bearer "` を返す現挙動。
    // api_key 空チェックは上位層 (呼び出し側) の責任という境界を明示。
    let header = build_whisper_authorization_header("");

    assert_eq!(header, "Bearer ");
}
```

### T5. build_whisper_authorization_header_does_not_trim_or_validate_input

**目的**: 前後 whitespace / multibyte api_key で trim/validation 一切しない現挙動の固定。

```rust
#[test]
fn build_whisper_authorization_header_does_not_trim_or_validate_input() {
    // ケース 1: 前後 whitespace は trim されず、Bearer の後ろにそのまま埋め込まれる
    {
        let header = build_whisper_authorization_header(" sk-xxx ");
        // " " (前) も " " (後) もそのまま保持される現挙動を固定。
        assert_eq!(header, "Bearer  sk-xxx ");
    }

    // ケース 2: multibyte api_key は UTF-8 panic を起こさず Bearer の後ろに埋め込まれる
    {
        let header = build_whisper_authorization_header("日本語キー");
        // multibyte char base 切り詰め回避と byte base パニック防止の両方を確認。
        assert_eq!(header, "Bearer 日本語キー");
    }
}
```
