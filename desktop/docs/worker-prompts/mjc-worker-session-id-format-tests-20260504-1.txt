## 追加するテスト 4 件 (関数名はそのまま使う)

### A. `start_assigns_unique_ids_across_consecutive_calls`
- 振る舞い: 同じ started_at で 3 回連続 `Session::start` を呼ぶと、3 つの id がすべて異なる (counter 単調増加保証)
- 不変条件: `s1.id != s2.id`, `s2.id != s3.id`, `s1.id != s3.id` (HashSet で重複なし確認でも可)
- 注意: グローバル `SESSION_COUNTER` を使うので、絶対値 (`"1000-0"` 等) は assert せず、**差異のみ** 確認する。並列 test 実行下でも安全。
- 設計意図: 同じ started_at で複数 session を作っても id 衝突しない不変条件 (履歴ファイル名衝突回避の根幹)。

### B. `start_id_starts_with_started_at_prefix_and_hyphen`
- 振る舞い: `Session::start("title".into(), 1700000000)` の id が `"1700000000-"` で始まる
- 不変条件: `session.id.starts_with("1700000000-")`、かつ `session.id.contains('-')` (format 契約の二重確認)
- 設計意図: session_store::parse_session_started_at_secs の入力契約 (id を `-` で split して `started_at_secs` を取り出す) を session 側で裏付ける。format が `"{seq}-{started_at}"` のような順序逆転に書き換えられたら即座に検知できる。

### C. `append_segment_after_finalize_records_post_finalize_segment`
- 振る舞い: `start` → `append` (1 件) → `finalize` → `append` (1 件) を実行すると、`segments.len() == 2` になる
- 不変条件:
  - `session.is_finalized() == true` (finalize 後の状態は変わらない)
  - `session.segments.len() == 2`
  - `session.segments[1].speaker == "post-finalize-speaker"` (finalize 後の append が確実に push される)
- 設計意図: 現状仕様 (finalize 後も append 可能) を test で固定する。将来「finalize 後の append は禁止」というリファクタが入った場合に気付ける。仕様変更が必要なら test を意図的に書き換える形で議論できる。

### D. `finalize_called_twice_overwrites_ended_at_with_latest_value`
- 振る舞い: `start("title", 1000)` → `finalize(1300)` → `finalize(1500)` を実行すると `ended_at == Some(1500)` になり、`duration_secs() == Some(500)` になる (後勝ち)
- 不変条件:
  - 1 回目 finalize 後: `ended_at == Some(1300)`, `duration_secs() == Some(300)`
  - 2 回目 finalize 後: `ended_at == Some(1500)`, `duration_secs() == Some(500)` (後勝ち上書き)
- 設計意図: 現状仕様 (finalize 冪等性なし、後勝ち) を test で固定する。realtime engine が「会議終了タイミングを更新する」ような場合、最新値が反映される動作の前提。
