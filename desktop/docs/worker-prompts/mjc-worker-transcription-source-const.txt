# 背景

前 mjc-main-16 Loop 2 (commit 9eecc11) で `src-tauri/src/audio_event.rs` に `pub(crate) const AUDIO_SOURCE_MICROPHONE = "microphone"` / `pub(crate) const AUDIO_SOURCE_SYSTEM_AUDIO = "system_audio"` を導入し、`src-tauri/src/audio.rs` / `src-tauri/src/system_audio.rs` の event payload 出力 4 箇所を集約済み。

ただし `src-tauri/src/transcription.rs` には同じ文字列値 `"microphone"` / `"system_audio"` が parse 処理 (CLI/設定入力解釈) として line 1052 / 1056 の match arm に残っている。これは event payload 出力責務 (audio_event.rs) とは異なる責務 (parse 処理) のため、audio_event.rs の定数を流用するのではなく **transcription.rs ローカルに別途定数定義する** 形が責務境界を保つ正しい設計選択。

これは前 mjc-main-16 SUMMARY で「同じ文字列値だが別責務」と明示された候補 G4 の 1 ループ完結スライス。

# Why (なぜこれが価値か)

1. **責務境界の明示**: audio_event.rs (event payload 出力) と transcription.rs (parse 処理) が同じ文字列値を持つことを「2 箇所定数定義」という形で目に見える分離にする。同じ文字列値だが違う責務であることをコードレベルで宣言する = DRY 違反だが「責務境界の見える化 > DRY」優先判断。
2. **typo 検知の強化**: 文字列リテラル → 定数化により、誤タイプによる match 失敗を compile error 化。
3. **将来の identifier 変更時の単一書き換え地点化**: parse 処理側の "microphone" / "system_audio" を変更したい場合、定数 1 箇所のみで対応可能 (CLI 後方互換ブレークの判断もしやすい)。
4. **Tidy First 原則**: 振る舞い不変、構造のみ改善、テスト追加のみ。前 mjc-main-16 Loop 2 と完全対称な 2 件目 application = 知見定着。
