# 追加する test 3 件 (Tidy First、ユーザー観測不可、振る舞い不変)

## test 1: `append_preserves_offset_secs_zero_for_immediate_segment_at_session_start`
- **目的**: SessionManager::append が offset_secs=0 (session 開始直後の即時 segment、左境界) を受け入れて Session.segments[0].timestamp_offset_secs に原値透過で保存する契約を CI 固定
- **手順**:
  1. `let manager = SessionManager::new();`
  2. `manager.start("instant_segment_session".into(), 1_700_000_000).expect("start");`
  3. `manager.append("Alice".into(), 0, "immediate".into()).expect("append with offset_secs=0 should succeed");`
  4. `let session = manager.finalize(1_700_000_100).expect("finalize");`
  5. `assert_eq!(session.segments.len(), 1, "1 segment が保存される");`
  6. `assert_eq!(session.segments[0].timestamp_offset_secs, 0, "offset_secs=0 は原値透過で保存される");`
  7. `assert_eq!(session.segments[0].text, "immediate", "text も原値透過");`
- **意図**: 「if offset_secs == 0 { skip }」のような『0 を unset として扱う』防衛的最適化誤改修を検知する装置。session 開始直後の発話 (offset_secs=0) は real flow で発生しうる。
- **assert メッセージ**: 「offset_secs=0 (session 開始直後の即時 segment) は原値透過で保存される。0 を unset として扱う防衛的最適化を CI で遮断する。」

## test 2: `append_preserves_offset_secs_u64_max_without_truncation`
- **目的**: SessionManager::append が offset_secs=u64::MAX (極端値、右境界) を truncation なく原値透過で Session.segments[0].timestamp_offset_secs に保存する契約を CI 固定
- **手順**:
  1. `let manager = SessionManager::new();`
  2. `manager.start("far_future_segment".into(), 1_700_000_000).expect("start");`
  3. `manager.append("Bob".into(), u64::MAX, "far future".into()).expect("append with offset_secs=u64::MAX should succeed");`
  4. `let session = manager.finalize(u64::MAX).expect("finalize");`
  5. `assert_eq!(session.segments[0].timestamp_offset_secs, u64::MAX, "offset_secs=u64::MAX は truncation なく保存される");`
- **意図**: 内部で `offset_secs as i64` cast 等を入れた場合に signed overflow で負値化するような誤改修を検知する装置。Loop 1 T3 (started_at u64::MAX) / Loop 2 T3 (ended_at u64::MAX) と対称的に offset_secs 側も u64 型契約を CI 固定。
- **assert メッセージ**: 「offset_secs=u64::MAX は truncation なく保存される。`as i64` cast 等の符号付き truncation を CI で遮断する。」

## test 3: `append_preserves_empty_text_and_empty_speaker_without_skipping_segment`
- **目的**: SessionManager::append が text="" / speaker="" の空文字列入力を skip せず、segment として保存する契約を CI 固定 (文字列軸の境界)
- **手順**:
  1. `let manager = SessionManager::new();`
  2. `manager.start("empty_string_session".into(), 1_700_000_000).expect("start");`
  3. `manager.append(String::new(), 0, String::new()).expect("append with empty speaker and empty text should succeed");`
  4. `manager.append("Alice".into(), 5, "second".into()).expect("second append");`
  5. `let session = manager.finalize(1_700_000_100).expect("finalize");`
  6. `assert_eq!(session.segments.len(), 2, "空文字列入力でも segment は skip されず保存される");`
  7. `assert_eq!(session.segments[0].speaker, "", "empty speaker は原値透過");`
  8. `assert_eq!(session.segments[0].text, "", "empty text は原値透過");`
  9. `assert_eq!(session.segments[1].speaker, "Alice", "後続 append は前 segment に影響なく保存される");`
- **意図**: 「if text.is_empty() { skip }」のような『空文字列を skip する』防衛的最適化誤改修を検知する装置。realtime 文字起こしで空 segment が発生する可能性 (例: silence chunk、partial result 確定前の commit) があり、それを skip すると segment 順序の整合性が壊れる。append の責務は「呼ばれたら必ず push」であり、filtering は呼び出し側の責務という設計契約を CI 固定。
- **assert メッセージ**: 「空文字列入力でも segment は skip されず保存される。append の責務は『呼ばれたら必ず push』。filtering は呼び出し側の責務という設計契約を CI で固定する。」
