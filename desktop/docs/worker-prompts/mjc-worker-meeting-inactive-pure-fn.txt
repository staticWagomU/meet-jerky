# 背景

meet-jerky は会議アプリ (Zoom/Teams/Google Meet 等) の起動 → 検知 → 通知 → イベント emit を `src-tauri/src/app_detection.rs` で行っている。`DetectionState.last_seen: HashMap<String, Instant>` で「bundle_id → 最後に検知された時刻」を保持しているが、現状「会議が終了した (= watched bundle が長時間検知されない)」という状態を取り扱う仕組みは存在しない。

これに対し、product-concept 優先度 8 (会議サービス別の最適化) と優先度 9 (録音状態の透明性) の chain として「会議終了の検知 → 録音/文字起こし停止のナッジ通知」を実装するのが本タスク全体の最終目標。

ただし本ループはその全体目標の **Tidy First の最小スライス**: アプリ統合せず、純粋関数 + test だけを追加する。アプリ統合 (handle_detection 側からの呼び出し / 状態追加 / 実機タイマー側への組み込み) は次ループに分離する。

# Why (なぜこれが価値か)

1. **既存 precedent との完全対称**: `should_warn_polling_stall` (app_detection.rs:153-174) は browser_url_callback の停滞検知装置として「純粋関数 + test 5 件 (line 1439-1460) → handle_browser_url_detection 側 (line 196-208) で AtomicU64 と組み合わせて呼び出し」という 2 段構成で実装済み。本タスクはこの構成を完全コピーして "meeting inactive 検知" 版を作る。

## 追加するもの

### (1) 純粋関数 `should_notify_meeting_inactive`

`should_warn_polling_stall` (line 153-174) の直下、`fn handle_browser_url_detection` (line 182) の前に挿入する。doc comment + cfg attr + 関数本体のスタイルを `should_warn_polling_stall` と完全対称に揃える。

シグネチャ案:
```rust
/// 監視中の会議アプリが長時間検知されない場合に「会議終了」通知を発火すべきかを判定する純粋関数。
/// `Some(elapsed)` は通知対象の経過秒数、`None` は通知不要。
#[cfg_attr(not(target_os = "macos"), allow(dead_code))]
fn should_notify_meeting_inactive(
    now_secs: u64,
    last_seen_secs: u64,
    last_notified_secs: u64,
    inactive_threshold_secs: u64,
    throttle_secs: u64,
) -> Option<u64> {
    if last_seen_secs == 0 {
        return None;
    }
    if now_secs <= last_seen_secs {
        return None;
    }
    let elapsed = now_secs - last_seen_secs;
    if elapsed < inactive_threshold_secs {
        return None;
    }
    if now_secs.saturating_sub(last_notified_secs) < throttle_secs {
        return None;
    }
    Some(elapsed)
}
```

判定ロジック:
- `last_seen_secs == 0` → None (一度も検知されていない = 通知対象外)
- `now_secs <= last_seen_secs` → None (時刻同期問題に対する保守的扱い)
- `elapsed = now_secs - last_seen_secs`
- `elapsed < inactive_threshold_secs` → None (まだアクティブ範囲内)
- `now_secs.saturating_sub(last_notified_secs) < throttle_secs` → None (連続通知抑止)
- それ以外 → Some(elapsed)

**注意**: `should_warn_polling_stall` は `elapsed <= expected_interval_secs * 3` で「正常範囲」を判定 (倍率 3) しているが、本関数は閾値を直接引数で受け取るので倍率不要。**あえて** "<" と "<=" の境界条件を `should_warn_polling_stall` と揃える必要はなく、上記のロジック (`< threshold`) を採用する。これは inactive 用途では「閾値ちょうどでも通知発火させたい」設計のため。

2. **既存 test 4 件を変更しない**: should_warn_polling_stall_* は触らない。直下に純粋追加のみ。

5. **handle_detection / DetectionState / 実機統合は touch しない**: 本ループのスコープ外。次ループで実施する。
