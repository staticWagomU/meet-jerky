目的:
TranscriptView の `MEETING_START_REQUEST_EVENT` listener 開始失敗を、cleanup の解除失敗と区別して明示的に UI/console へ出し、開始していない listener を cleanup で解除しないようにしてください。

背景:
- Loop 217/218 で App / RingLightWindow / LiveCaptionWindow の listener 開始失敗は明示化済みです。
- `src/routes/TranscriptView.tsx` の会議開始要求 listener はまだ `listen(...).then(unlisten => unlisten()).catch(...)` cleanup 形で、listen 開始そのものが失敗した場合に「受信解除に失敗」として見える可能性があります。
- 会議検知からの録音開始要求が受け取れない状態は、録音状態の透明性と復旧性に関わります。

実装方針:
- `MEETING_START_REQUEST_EVENT` の `listen` に開始失敗用 `.catch(...)` を追加してください。
- 開始失敗時は `toErrorMessage(e)` で console.error し、`setMeetingError` にユーザーが分かる日本語エラーを入れてください。
- `.catch(...)` は `null` を返し、cleanup は `unlisten?.()` などで未開始 listener を解除しないようにしてください。
- `disposed` guard と `readPendingMeetingStartRequest()` / `setHasPendingMeetingStartRequest(true)` の既存挙動は維持してください。
- event 名、localStorage key、録音開始ロジック、UI 文言の大きな変更、他 listener、別ファイルのリファクタ、依存関係追加は禁止です。
