## j.php URL 仕様 (招待メール旧形式)

形式: `https://<site>.webex.com/<site>/j.php?MTID=<token>`

例:
- `https://acme.webex.com/acme/j.php?MTID=mabc123` (subdomain `acme`、path `/acme/j.php`)
- `https://webex.com/webex/j.php?MTID=mxyz` (root `webex.com`、path `/webex/j.php`)

検知条件 (誤検知防止のため厳格):
1. `is_webex_host(host)` で host が `*.webex.com` または `webex.com` であること (DNS label 検証付き、既存)
2. path が `/<single-segment>/j.php` または `/<single-segment>/j.php/` 形式であること
3. query に `MTID=<non-empty value>` が含まれること

「MTID 値が非空」の判定が新規必要。既存の `query_has_param(query, key, value)` は値完全一致なので使えない。
新規ヘルパー `query_has_non_empty_param(query: Option<&str>, key: &str) -> bool` を追加するか、
既存ヘルパーを再利用しつつ別の path 解析関数 `is_webex_jphp_meeting_url(host, path, query) -> bool` を追加してください。
最小実装で構造一貫性を優先 (`query_has_param` の symmetric な命名で `query_has_non_empty_param` 追加が妥当)。

## 実装方針 (TDD Red→Green)

2. **TDD Green**: 最小実装で pass させる
   - `query_has_non_empty_param` ヘルパー追加 (または等価関数)
   - `is_webex_jphp_meeting_url(host, path, query) -> bool` 追加

3. **追加テスト** (Red phase 後、Green phase 前に揃える):
   - **accept**: subdomain webex (`https://acme.webex.com/acme/j.php?MTID=mabc123`)
   - **accept**: root webex (`https://webex.com/webex/j.php?MTID=mxyz`)
   - **accept**: trailing slash 許容 (`https://acme.webex.com/acme/j.php/?MTID=mxyz`) ※既存パターンとの整合性のため、要否判断は実装者に委ねる。Personal Room (`/meet/<id>`) と同様 trailing slash 許容なら追加
   - **reject**: MTID query 欠如 (`https://acme.webex.com/acme/j.php`)
   - **reject**: MTID 空値 (`https://acme.webex.com/acme/j.php?MTID=`)
   - **reject**: 非 webex host (`https://fake-webex.example.com/acme/j.php?MTID=mxyz`)
   - **reject**: path segment 数違反 (例: `/acme/foo/j.php?MTID=mxyz` や `/j.php?MTID=mxyz`)
4. **rustdoc 分類ルール更新**: `is_webex_meeting_url` 上のコメントを以下に書き換え:
