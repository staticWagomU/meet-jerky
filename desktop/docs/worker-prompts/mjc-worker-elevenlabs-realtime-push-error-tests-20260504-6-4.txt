# 背景 / Why

`mod ws_task` 内の `fn push_error` (L428-L442) は WebSocket エラーイベントから
TranscriptionSegment を生成する核心 fn ですが、現状は `handle_event` 経由の
**間接テストのみ** (`scribe_error_events_are_queued_as_error_segments` L675、ファイル
トップ mod tests 内) でカバーされています。

直近のループ (commit d002aac) で **openai_realtime.rs 側の同種 push_error を 5 件で
直接テスト化** しました。本ループは elevenlabs 側を同じ 5 件構造で直接テスト化し、
両 Realtime engine の error path 不変条件を完全対称な水準で保護します。

直接テストを追加する価値:
1. **prefix 文言契約の固定**: `"[ElevenLabs Realtime エラー: {message}]"` の prefix
   `"[ElevenLabs Realtime エラー: "` と suffix `"]"` をユーザー向け文言として固定。
2. **不変条件の固定**: `start_ms=0, end_ms=0, is_error=Some(true)` の 3 値を 1 関数で
   契約強制。
3. **source/speaker passthrough 契約**: `Some(TranscriptionSource::Microphone)` / `None` の
   双方向 passthrough と speaker の clone 動作を直接固定。
4. **関数分離リファクタへの耐性**.
5. **openai_realtime との完全対称**: prefix 文言の違いを除いて構造的に同じテストを書く。

# 配置方針 (重要)

elevenlabs_realtime.rs では既に **`mod ws_task` 内に `mod pending_timeout_tests`**
(L482-L631) があり、ws_task 内 private fn の直接テスト場として使われています
(前セッション ループ 3 で is_scribe_error_event / extract_transcript /
extract_error_message の 6 件をここに追加済み)。

本ループは **`mod pending_timeout_tests` の末尾**に追加してください。
visibility を広げる必要はありません。`super::push_error` で呼べます。

ファイルトップの `mod tests` (L634-) には追加しないこと。

# 具体的な実装変更

実装は変更しません。**`mod pending_timeout_tests` (L482-L631) の末尾**
(`extract_error_message_traverses_three_priority_paths` の終了 `}` の直後、
`}` (L631 の inner mod 閉じ) の直前) に以下の 5 件を追加してください。

各テストは独立した `#[test] fn` とすること。

inner mod の use はインデント 8 空白。`super::*;` ではなく `super::push_error` 形式で呼ぶ
(既存 `super::is_scribe_error_event(...)` と同流儀)。

inner mod 内では `Arc`, `Mutex`, `Vec`, `Some`, `Option` 等は標準で使えるが、
`TranscriptionSegment`, `TranscriptionSource` は `super::super::TranscriptionSegment` または
fully qualified path で参照する必要がある。**既存テスト** (L549- 等) で `super::is_scribe_error_event`
の他に `Value` 等が使われている箇所を参照して、同流儀で書くこと。

具体的には、既存テスト `extract_error_message_traverses_three_priority_paths` (L608) などが
どの fully qualified path を使っているか確認した上で、`Arc<Mutex<Vec<TranscriptionSegment>>>` の
import 方法を踏襲してください。

### T1. push_error_format_prefix_and_suffix_are_fixed

```rust
#[test]
fn push_error_format_prefix_and_suffix_are_fixed() {
    use std::sync::Arc;
    use parking_lot::Mutex;
    let pending: Arc<Mutex<Vec<super::super::TranscriptionSegment>>> =
        Arc::new(Mutex::new(Vec::new()));
    let speaker = Some("自分".to_string());

    super::push_error(
        &pending,
        &speaker,
        Some(super::super::TranscriptionSource::Microphone),
        "connection closed".to_string(),
    );

    let segments = pending.lock();
    assert_eq!(segments.len(), 1);
    // prefix と suffix を含む完全一致で文言契約を強制。
    assert_eq!(
        segments[0].text,
        "[ElevenLabs Realtime エラー: connection closed]"
    );
}
```

### T3. push_error_passes_source_through

```rust
#[test]
fn push_error_passes_source_through() {
    use std::sync::Arc;
    use parking_lot::Mutex;

    // ケース 1: Some(Microphone) は Some(Microphone) のまま渡る
    {
        let pending: Arc<Mutex<Vec<super::super::TranscriptionSegment>>> =
            Arc::new(Mutex::new(Vec::new()));
        let speaker = Some("自分".to_string());
        super::push_error(
            &pending,
            &speaker,
            Some(super::super::TranscriptionSource::Microphone),
            "err".to_string(),
        );
        assert_eq!(
            pending.lock()[0].source,
            Some(super::super::TranscriptionSource::Microphone)
        );
    }

    // ケース 2: None は None のまま渡る
    {
        let pending: Arc<Mutex<Vec<super::super::TranscriptionSegment>>> =
            Arc::new(Mutex::new(Vec::new()));
        let speaker = Some("自分".to_string());
        super::push_error(&pending, &speaker, None, "err".to_string());
        assert_eq!(pending.lock()[0].source, None);
    }
}
```

# 実装上の注意

- **配置場所**: `mod pending_timeout_tests` (L482-L631) の末尾、`}` (L631 の inner mod 閉じ) の直前。
- **inner mod のため、各テストで `use std::sync::Arc; use parking_lot::Mutex;` を fn 内で書く**
  (既存スタイルに合わせる。inner mod 全体の `use` を 1 つだけ追加してもよいが、既存 6 件の追加 fn が
  関数内 use を使っていない場合は `super::super::TranscriptionSegment` 形式に統一する)。
- **既存 6 件のテストがどう書いているかを必ず確認**してから貼り付け方を調整すること。
  特に `Arc<Mutex<Vec<TranscriptionSegment>>>` の path 解決方法は、既存テストを 1 つ読んで踏襲する。
  もし既存テストが inner mod の **トップ** で `use super::super::*;` のような import を
  していたら、その流儀に従って fn 内 use を不要にする。
- 実装 (`fn push_error`) は変更しない。visibility 変更しない。
- 他のテスト関数を変更しない。
- ファイルトップの `mod tests` (L634-) には追加しない。
- `#[allow(dead_code)]` 等の attribute 追加不要。
