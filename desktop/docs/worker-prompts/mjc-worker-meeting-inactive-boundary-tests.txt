# 背景

直前のループ (commit e969943) で `should_notify_meeting_inactive` 純粋関数 + 基本 test 5 件 (T1-T5) を `src-tauri/src/app_detection.rs` に追加した。これは会議終了検知 (product-concept 優先度 8/9) の Loop 1 完結スライス。

T1-T5 で「初回 / アクティブ範囲内 / 通知発火 / throttle 抑止 / 同時刻保守」の 5 状態を網羅したが、**境界条件 (boundary)** はまだ固定していない。具体的には:

- 設計判断: `elapsed < inactive_threshold_secs` 境界 (`<=` ではなく `<`) = 「閾値ちょうどの elapsed = 通知発火」 (前 mjc-main-17 Loop 1 の commit 説明より)
- u64 の overflow / saturating 動作 (now_secs = u64::MAX 等)
- 引数 0 値 (inactive_threshold_secs=0 / throttle_secs=0)
- last_notified > now_secs (時刻巻き戻し / saturating_sub の 0 飽和)

これらが test で固定されていないと、将来的な誤改修 (`<=` への変更、saturating_sub の panic 化等) を検知できない。

# Why (なぜこれが価値か)

1. **設計判断の可視化**: Loop 1 commit 説明で「`elapsed < threshold` を採用」と明記したが、test がないと将来「説明と異なる」改修を防げない。境界 1 個 (T_boundary) を test 化することで設計意図を仕様レベルに格上げする。
2. **saturating の非 panic 契約固定**: `now_secs.saturating_sub(last_notified_secs)` は Loop 1 で導入した saturating ガード。`last_notified > now_secs` (時計巻き戻し) で panic 化への誤改修を T_clock_skew で検知する。
3. **0 値 boundary の挙動明示**: inactive_threshold_secs=0 / throttle_secs=0 は実用設定としてはあり得ないが、関数の数学的契約として「閾値 0 → 即発火」「throttle 0 → 連続発火」が満たされていることを T_zero_threshold / T_zero_throttle で固定する。
4. **u64 overflow boundary**: now_secs=u64::MAX で elapsed = u64::MAX - last_seen が巨大になっても Some(elapsed) を返す = panic 化への誤改修検知。
5. **Loop 1 と完全対称な構造**: 同じファイル / 同じ関数 / 直下追加 = worker が 1 ターンで完走する確実な scope。

## 実装上の注意

1. **T6 が最重要**: 設計判断 1 (`<` boundary) の固定。assert は `Some(300)` (threshold と elapsed が同値) で発火することを必ず明示。
2. **T9 の saturating 動作確認**:
   - `now_secs.saturating_sub(last_notified_secs)` は last_notified > now_secs のとき 0 を返す
   - 0 < throttle (600) なので throttle check に引っかかり None
   - **重要**: T9 では elapsed = 400 ≥ threshold 300 で第 4 条件は通過するが、第 5 条件 (throttle) で None になる経路
3. **T10 の `u64::MAX - 1`**: assert に `Some(u64::MAX - 1)` を直接書く。リテラル化禁止 (= `Some(18446744073709551614)` は書かない)。
4. **既存 test 5 件 (T1-T5) を変更しない**: 直下に純粋追加のみ。
5. **import 追加禁止 / module 分割禁止 / 関数本体 (line 176-201) も touch しない**: tests mod 内のみ。
