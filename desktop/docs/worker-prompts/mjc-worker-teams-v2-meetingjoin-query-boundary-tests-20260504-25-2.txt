# 追加する test 3 件 (具体仕様)

## 背景
- `classify_meeting_url` は Microsoft Teams `/v2` 経路で `is_teams_meeting_url` (l.769) → `query_has_param(query, "meetingjoin", "true")` (l.776) を呼ぶ。
- `query_has_param` (l.801) は `query.split('&').any(|param| { let (k, v) = param.split_once('=').unwrap_or((param, "")); k.eq_ignore_ascii_case(key) && v.eq_ignore_ascii_case(value) })` で実装。
- 既存 test:
  - l.1208 = `?meetingjoin=true&context=secret` (meetingjoin **先頭**) accept ✅
  - l.1215 = `?MEETINGJOIN=TRUE#meeting` (大小区別なし) accept ✅
  - l.1222/1252 = `?meetingjoin=true` (単独) accept ✅
  - l.1396/1429 = `?meetingjoin=false` reject (value 違反) ✅
- 未保護の境界:
  - meetingjoin が **中間** (`?context=x&meetingjoin=true`) accept = `query.split('&').any(...)` の 2 番目以降イテレーションマッチ動作の executable spec
  - **重複 key** (`?meetingjoin=false&meetingjoin=true`) accept = `any()` の one of マッチで true 動作
  - **value 部分なし** (`?meetingjoin`) reject = `split_once('=').unwrap_or((param, ""))` で param_value="" となり "true" と一致しない動作
- これら 3 つは「対称的補強の 3 軸構造」に綺麗に適用可能 = 「会議検知の信頼性」軸 (優先順位 #2) への直接寄与。

## 追加 test 仕様 (mod tests 末尾、Loop 1 の last test の直後に追加)

### T1: `classify_meeting_url_accepts_teams_v2_with_meetingjoin_true_in_middle_param_position`
- 目的: `query.split('&').any(...)` の中間 param マッチ動作を CI 固定。「first param のみ check」のような最適化誤改修を遮断。
- 仕様: `classify_meeting_url("https://teams.microsoft.com/v2/?context=secret&meetingjoin=true")` が `Some(MeetingUrlClassification { service: "Microsoft Teams".to_string(), host: "teams.microsoft.com".to_string() })` を返すことを assert。
- 既存 l.1208 (`?meetingjoin=true&context=secret`) は meetingjoin が先頭 = 1 番目のイテレーションでマッチ = 中間 param マッチは未保護。
- assert メッセージ: "Teams /v2 経路の query_has_param は & 区切りで全 param を走査する契約 (現契約: query.split('&').any(...)) = meetingjoin が 2 番目以降の位置にあっても accept される必要がある"

### T2: `classify_meeting_url_accepts_teams_v2_with_duplicate_meetingjoin_keys_when_one_matches`
- 目的: `any()` の one-of マッチ動作 (重複 key の 1 つだけ true でも全体 true) を CI 固定。「重複 key を reject する厳格化」誤改修を遮断。
- 仕様: `classify_meeting_url("https://teams.microsoft.com/v2/?meetingjoin=false&meetingjoin=true")` が `Some(MeetingUrlClassification { service: "Microsoft Teams".to_string(), host: "teams.microsoft.com".to_string() })` を返すことを assert。
- assert メッセージ: "Teams /v2 経路の query_has_param は重複 key の場合 1 つでもマッチで true を返す契約 (現契約: query.split('&').any(...)) = meetingjoin=false&meetingjoin=true で後者がマッチして accept される必要がある"

### T3: `classify_meeting_url_rejects_teams_v2_with_meetingjoin_key_only_no_equals`
- 目的: `split_once('=').unwrap_or((param, ""))` の value 部分なし時の挙動 (param_value="") を CI 固定。「value 部分なしを true 扱いする寛容化」誤改修を遮断。
- 仕様: `classify_meeting_url("https://teams.microsoft.com/v2/?meetingjoin")` が `None` を返すことを assert。
- 詳細: `split_once('=')` が None → unwrap_or = ("meetingjoin", "") → param_key match だが param_value="" != "true" → false → any() = false → query_has_param = false → is_teams_meeting_url = false → classify = None。
- assert メッセージ: "Teams /v2 経路の query_has_param は = が無い param を value=\"\" として扱う契約 (現契約: split_once('=').unwrap_or((param, \"\"))) = meetingjoin (= なし) は value=\"\" となり \"true\" と一致せず reject される必要がある"
