# 追加する test 3 件 (具体仕様)

## 背景
- `classify_meeting_url` は内部で `is_google_meet_code_path(path: &str) -> bool` を呼ぶ。
- `is_google_meet_code_path` (l.719) は path から `/` prefix と `/` suffix を strip した後、`-` 区切り 3 セグメント (first / second / third) に分割し、それぞれ `has_ascii_lowercase_len(first, 3)` / `(second, 4)` / `(third, 3)` で桁数 (3-4-3) と ASCII 小文字を check する。
- 既存 test では `/abc-defg-hij` (3-4-3 accept、l.1017-) / `/ABC-defg-hij` (大文字混入 reject、l.1346) / `/abc-defg` (区切り 1 つ短い reject、l.1342) はカバーされているが、**「区切り `-` は正しく 2 つあるが各セグメント桁数だけ違反する」境界 reject は完全未保護**。
- Google Meet の公式 URL 形式は `xxx-yyyy-zzz` (3-4-3) で固定 = 桁数 1 つでも違えば偽 URL = 「会議検知の信頼性」(優先順位 #2) への直接寄与。

## 追加 test 仕様 (mod tests 末尾、`classify_meeting_window_title_rejects_control_characters` の直後に追加)

### T1: `classify_meeting_url_rejects_google_meet_code_path_with_short_first_segment`
- 目的: first segment が 2 桁 (3 桁未満、左境界外側) の path を reject する現契約を CI 固定。`has_ascii_lowercase_len(first, 3)` の桁契約を保護。
- 仕様: `classify_meeting_url("https://meet.google.com/ab-defg-hij")` が `None` を返すことを assert。
- assert メッセージ (要 5 引数版): "Google Meet path code は first segment 3 桁が必須 (現契約: has_ascii_lowercase_len(first, 3)) = 2 桁 (左境界外側) は reject される必要がある"
