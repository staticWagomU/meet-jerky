## 2. `src/utils/audioDropCountPayload.ts` を新規作成
既存の `src/utils/audioLevelPayload.ts` (内容は本プロンプトのコメントで再掲する) と完全対称な構造で書く。

```ts
import type { AudioDropCountPayload } from "../types";

function isAudioDropCountSource(
  value: unknown,
): value is AudioDropCountPayload["source"] {
  return value === "microphone" || value === "system_audio";
}

export function isAudioDropCountPayload(
  value: unknown,
): value is AudioDropCountPayload {
  if (!value || typeof value !== "object") {
    return false;
  }
  const candidate = value as Partial<AudioDropCountPayload>;
  return (
    isAudioDropCountSource(candidate.source) &&
    typeof candidate.dropped === "number" &&
    Number.isFinite(candidate.dropped) &&
    candidate.dropped >= 0
  );
}
```

- `dropped >= 0` チェックを追加: rust 側 usize は非負だが JS では負数や Number.NaN が来る防御的検査。`Number.isFinite` で NaN/Infinity も弾く (audio-level の `Number.isFinite(candidate.level)` と対称)。
- `>= 0` 比較は `Number.isFinite` 通過後なので NaN による比較演算の罠は無い。

```ts
// Route audio-drop-count events by source (cumulative)
useEffect(() => {
  let disposed = false;
  const unlistenPromise = listen<unknown>("audio-drop-count", (event) => {
    if (disposed) {
      return;
    }
    const payload = event.payload;
    if (!isAudioDropCountPayload(payload)) {
      setAudioDropCountListenerError("音声 drop 通知の形式が不正です。");
      return;
    }
    setAudioDropCountListenerError(null);
    console.warn(
      `[audio-drop-count] ${payload.source} で ${payload.dropped} sample 破棄`,
    );
    if (payload.source === "microphone") {
      setMicrophoneDropCountTotal((prev) => prev + payload.dropped);
    } else if (payload.source === "system_audio") {
      setSystemAudioDropCountTotal((prev) => prev + payload.dropped);
    }
  })
    .then((unlisten) => {
      if (!disposed) {
        setAudioDropCountListenerError(null);
      }
      return unlisten;
    })
    .catch((e) => {
      if (!disposed) {
        const msg = toErrorMessage(e);
        console.error("音声 drop 監視の開始に失敗しました:", msg);
        setAudioDropCountListenerError(
          `音声 drop 監視の開始に失敗しました: ${msg}`,
        );
      }
      return null;
    });

  return () => {
    disposed = true;
    unlistenPromise
      .then((unlisten) => unlisten?.())
      .catch((e) => {
        console.error("音声 drop 監視の解除に失敗しました:", toErrorMessage(e));
      });
  };
}, []);
```

# 実装上の注意 (踏みやすい罠)
- **既存の "audio-level" useEffect (line 754-799) と完全対称な構造**にすること。ハンドラ命名 / unlistenPromise / catch / cleanup の 4 ブロック構造をそのまま踏襲。新しいパターンを発明しない。
- **state は累積 (cumulative)**。`prev + payload.dropped` で増分加算。本ループでは reset しない (UI 表示と reset 戦略は次ループ送り)。
- **TypeScript strict mode** + `noUnusedLocals: true` (tsconfig.json line 19) のため、宣言した state や setter は **必ず参照すること**。state は useEffect の setter で参照され、setter は handler 内で呼ばれるので OK。`microphoneDropCountTotal` / `systemAudioDropCountTotal` 自体は本ループで read しないが、state declaration として最低限必要。**`noUnusedLocals` で warning が出る場合は、useEffect 内で referenced されている setter のみで十分なはず**。もし unused warning が出るなら、 `// eslint-disable-next-line` ではなく **state 自体を読み出す箇所を 1 つでも追加** すること (例: console.log の中で `${microphoneDropCountTotal}` を出す、など。最小コードで warning を回避)。
- **`setAudioDropCountListenerError` も同様**。本ループでは error UI 表示しないが、setter は handler 内で呼ばれている。state 自体は read しなくて OK だが、TS strict が unused としてエラーにする場合は console.warn の引数等で 1 か所参照する。
- **state 名は明示的に `Total` 接尾辞**: 累積であることを名前で示す。次ループで `microphoneDropCount` (per-event) と `microphoneDropCountTotal` (cumulative) を区別する余地を残す。
- **既存 `console.warn` パターンに沿う**: `console.warn` で onset 通知、`console.error` で listener 失敗通知。既存 audio-level の `console.error("音声レベル監視の開始に失敗しました:")` と対称。
- **Tauri event 名 "audio-drop-count"** は rust 側 `app_handle.emit("audio-drop-count", ...)` (audio.rs line 401, system_audio.rs line 333) と完全一致する文字列リテラル。タイポしない。
- **isAudioDropCountPayload の `>= 0` 厳格化**: rust 側 usize は非負保証だが、frontend 側で防御的に弾くことで "rust 側のバグで負数が来た場合に UI が壊れない" 契約を確立する。Number.isFinite 後なので NaN と +/-Infinity は通らない。
- **css / className の追加禁止**。本ループは pure logic 追加。スタイル変更は次ループ送り。
