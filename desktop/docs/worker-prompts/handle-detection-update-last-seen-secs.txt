## 実装上の注意

### コード追加例
既存 throttle ブロック (line 122-130 付近の `{ let mut last_seen = ... }`) の **直後** に挿入:

```rust
// `should_notify_meeting_inactive` 用に epoch secs ベースの最終検知時刻を更新する
// (案 A 二重管理: 既存 Instant ベース throttle と並走、Loop 3 で wrapper 関数が読み取る予定)。
{
    let now_secs = std::time::SystemTime::now()
        .duration_since(std::time::UNIX_EPOCH)
        .map(|d| d.as_secs())
        .unwrap_or(0);
    state
        .last_seen_secs
        .lock()
        .insert(bundle_id.to_string(), now_secs);
}
```

### SystemTime パターンの再利用
- `std::time::SystemTime::now().duration_since(std::time::UNIX_EPOCH).map(|d| d.as_secs()).unwrap_or(0)` は既存 line 221-224 / 244-248 と同じ書き方を踏襲
- `unwrap_or(0)`: pre-1970 clock skew 時に 0 を返す = `should_notify_meeting_inactive` の `last_seen_secs == 0` ガードと整合 (= 通知しない、保守的)
- `use std::time::SystemTime` の追加 import は不要 (既存 `use std::time::{Duration, Instant};` line 22 のままで `std::time::SystemTime` は fully-qualified path で使用可能 = 既存パターン踏襲)

### Lock 順序
- 新規 `state.last_seen_secs.lock()` は既存 `state.last_seen.lock()` (= line 123 で取得済の MutexGuard) の **drop 後** に取得 = 既存 throttle ブロック `{ ... }` の閉じ括弧の **後** に挿入することで guarantee
- これにより既存 lock 取得順序 (last_seen → latest_payload) を干渉せず、新規 last_seen_secs は独立した critical section
- デッドロック回避
