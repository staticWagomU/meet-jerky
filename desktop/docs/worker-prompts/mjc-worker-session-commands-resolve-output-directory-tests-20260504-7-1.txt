## 1. 背景 / Why
session_commands.rs (399 行 / 13 件既存) は `*_inner` パターンでテスト可能設計になっており、
主要な start/finalize/discard/list の正常系・異常系は既存テストで覆われている。
ただし以下の 2 fn が直接テストでカバーされていない:

- `resolve_output_directory(settings_state)` (private fn, line 34-40):
  AppSettings.output_directory が None or Some("") なら `default_output_directory()` を返し、
  Some(非空文字) ならその文字列を `PathBuf::from(...)` で wrap して返す。
  この 3 ケースは `match Some(dir) if !dir.is_empty()` の guard を含む 1 行の制御フロー全体の契約。
  validation/trim は一切しない上位層責任境界 fn (前セッション ループ 2 で確立した契約強制パターン
  の対象として典型的)。
- `default_offset()` (private fn, line 28-30):
  JST 固定 = `FixedOffset::east_opt(9 * 3600)` のハードコード値。
  「将来ユーザー設定化する際はここを差し替えれば良い」とコメントあるため、
  現状は単一定数契約として固定すべき。9 → 8 や 9*3600 → 9*60 のような typo 回帰検知装置。

これら 5 件は session_commands.rs 内の private fn 直接テスト化で完結し、
SessionManager や session_store, tauri::State などの依存を引き込まない。
SettingsStateHandle は `pub Mutex<AppSettings>` で public field、AppSettings は Default 実装あり、
テストから `SettingsStateHandle(parking_lot::Mutex::new(AppSettings { output_directory: ..., ..AppSettings::default() }))` で直接構築可能。
**`SettingsStateHandle::new()` は呼ばないこと**(`AppSettings::load()` 経由で実環境の
~/Library/Application Support/meet-jerky/settings.json を読みに行くため、テスト副作用と並列実行不安定の元になる)。

## 2. 具体的 test 案 + assertion レベル
src-tauri/src/session_commands.rs の `mod tests` 末尾 (line 398 の `}` 直前) に以下 5 件を追加する。

| # | 関数名 | 検証内容 |
|---|--------|---------|
| T1 | resolve_output_directory_returns_default_when_setting_is_none | AppSettings { output_directory: None, ..default() } で SettingsStateHandle 構築 → default_output_directory() と等しい PathBuf を返す。assert_eq! で比較。 |
| T2 | resolve_output_directory_returns_default_when_setting_is_empty | output_directory: Some(String::new()) でも T1 と同じ結果 → guard `if !dir.is_empty()` の削除リファクタを検知。 |
| T3 | resolve_output_directory_returns_setting_path_when_set_to_nonempty | output_directory: Some("/tmp/custom-output".into()) → PathBuf::from("/tmp/custom-output") と一致 (assert_eq!)。default_output_directory() とは「異なる」ことも assert_ne! で確認。 |
| T4 | resolve_output_directory_does_not_trim_whitespace_in_setting_path | output_directory: Some(" /tmp/path ".into()) (前後 whitespace 込み) → PathBuf::from(" /tmp/path ") そのまま (assert_eq!)。**前後 whitespace は trim しない現挙動を契約化** (上位層責任境界パターン)。 |
| T5 | default_offset_returns_jst_offset_constant | default_offset() == FixedOffset::east_opt(9 * 3600).unwrap() を assert_eq!。さらに `.local_minus_utc() == 9 * 3600` も追加で assert_eq! して秒値を二重に固定 (9 を 8 に書き換える typo + 9*3600 を 9*60 に書き換える typo を別軸で検知)。 |

各 test は #[test] 属性、tokio など async 不要。SettingsStateHandle 構築は次のヘルパで簡略化してよい:

```rust
fn build_settings_handle_with_dir(dir: Option<String>) -> SettingsStateHandle {
    use crate::settings::AppSettings;
    use parking_lot::Mutex;
    SettingsStateHandle(Mutex::new(AppSettings {
        output_directory: dir,
        ..AppSettings::default()
    }))
}
```

ただし、ヘルパを 1 つだけ追加するならシグネチャはシンプルに、追加コード量を最小に保つこと。
2 種類以上のヘルパが必要になるなら、各 test の冒頭で直接 SettingsStateHandle を構築する方が読みやすい。判断は worker に任せる。

## 3. 実装上の注意
- `use parking_lot::Mutex;` と `use crate::settings::AppSettings;` は tests mod の中で `use super::*;` の後に追加する。既存 use との重複に注意。
- `default_output_directory()` は `dirs::data_dir()` 依存で実環境のパスを返すが、`PathBuf` 比較は値比較なので副作用なし。CI / ローカル両方で動く。
- T4 の whitespace 込みパスは「現在の挙動」を固定するもの。将来 trim を追加する場合に test 修正が必要、というのが意図 (誤リファクタ検知装置)。
- `resolve_output_directory` は `&SettingsStateHandle` を引数に取るので、`&handle` を渡す。
- 既存 13 件のテスト名・本体は **絶対に変更しない**。追加のみ。
- `AppSettings { ..default() }` パターンは `..AppSettings::default()` と書く必要がある (struct update syntax)。
- 5 件すべて test 関数のみ追加。session_commands.rs の本体ロジック (line 1-153) は **一切変更しない**。
- `chrono::FixedOffset` は line 10 ですでに import 済み。tests mod 内では `use super::*;` で参照可能。
