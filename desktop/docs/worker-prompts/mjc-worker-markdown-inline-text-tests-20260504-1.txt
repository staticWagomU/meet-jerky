## 前提情報 (調査済みなので追加調査は最小限)
- `markdown.rs` の private fn `inline_markdown_text` (L51-61):
  ```rust
  fn inline_markdown_text(value: &str) -> String {
      let normalized = value.split_whitespace().collect::<Vec<_>>().join(" ");
      let mut escaped = String::with_capacity(normalized.len());
      for ch in normalized.chars() {
          if matches!(ch, '\\' | '`' | '*' | '_' | '[' | ']') {
              escaped.push('\\');
          }
          escaped.push(ch);
      }
      escaped
  }
  ```
- エスケープ対象は 6 文字: `\\`, `` ` ``, `*`, `_`, `[`, `]`
- 既存テスト `test_format_session_markdown_escapes_inline_markdown_marks` (L124) は `*`, `[`, `]`, `` ` ``, `\\` をカバーするが、**`_` (アンダースコア) のエスケープは未カバー** — これが致命的なリグレッション検知漏れ。
- `mod tests { use super::*; }` 既存なので private fn は `super::inline_markdown_text` で参照できる。

## 追加するテスト 5 件 (関数名はそのまま使う)

### A. `inline_markdown_text_returns_empty_for_empty_input`
- 振る舞い: `inline_markdown_text("") == ""`
- 設計意図: `split_whitespace().join(" ")` の挙動 (空入力 → 空 vec → "") を仕様として固定。

### B. `inline_markdown_text_returns_empty_for_whitespace_only_input`
- 振る舞い: 以下 3 assertion
  - `inline_markdown_text("   ") == ""`
  - `inline_markdown_text("\t\n  ") == ""`
  - `inline_markdown_text(" \t \n ") == ""`
- 設計意図: whitespace のみは出力ゼロ。`split_whitespace` が tab/newline を含むすべての ASCII whitespace を区切り文字として扱う仕様の裏付け。

### C. `inline_markdown_text_escapes_underscore_explicitly`
- 振る舞い:
  - `inline_markdown_text("_") == "\\_"`
  - `inline_markdown_text("hello_world") == "hello\\_world"`
  - `inline_markdown_text("__bold__") == "\\_\\_bold\\_\\_"`
- 設計意図: **既存テストで `_` のエスケープが一度も確認されていない致命的な検知漏れを補完**。`_` を escape 対象から外すリファクタが入った場合に即座に検知。

### D. `inline_markdown_text_escapes_each_special_char_individually`
- 振る舞い: 6 文字を個別 1 文字 + 単独 escape 確認
  - `inline_markdown_text("\\") == "\\\\"`
  - `inline_markdown_text("`") == "\\`"`
  - `inline_markdown_text("*") == "\\*"`
  - `inline_markdown_text("_") == "\\_"` (※ C と重複するが、6 文字を網羅する文書として残す)
  - `inline_markdown_text("[") == "\\["`
  - `inline_markdown_text("]") == "\\]"`
- 設計意図: エスケープ対象 6 文字すべてが個別に確認される。将来エスケープ対象を 1 文字でも変更したら即座に検知。

### E. `format_session_markdown_handles_single_segment_correctly`
- 振る舞い:
  - meta `title="会議"`, `started_at_display="2026-04-17 14:30"`
  - segments 1 件: `speaker="自分"`, `timestamp_display="14:30:05"`, `text="hello"`
  - 期待: `"# 会議 - 2026-04-17 14:30\n\n**[14:30:05] 自分:** hello  "`
- 設計意図: 既存 happy_path は 2 segments の場合のみ。1 segment の場合に `lines.join("\n")` がどう振る舞うかを test で固定。
