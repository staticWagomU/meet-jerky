- そのため handoff prompt は依然として **tmux 引数長制限の影響を受ける**
- 対策 = 「ファイル参照型 handoff prompt」**precedent を継承**: 起動 prompt は短縮 + フル handoff 内容は別ファイル (`docs/handoff/mjc-main-YYYYMMDD-N.txt`) に書き、起動 prompt 内で「Read で完全に読め」と指示する (本 prompt と同パターン、連続 41 セッション目)

## worker prompt 必須要素 (100+ 連続ループで実証済、継続適用、7 つ全て明示)

worker prompt には以下を **必ず** 含める:
1. **冒頭で「AGENT_LOG.md の末尾 350 行を読め」**
2. **「AGENT_LOG.md は時系列順 = 最古ログが先頭、最新ログが末尾。新規追記は必ずファイル末尾に行う」**
3. **「先頭は絶対に触らない」**
4. **具体手順**: `tail -10 AGENT_LOG.md` で末尾を確認 → 末尾の `---` 直後に追記
5. **規模超過防止段落** = 担当範囲外の編集禁止 + test 件数の上限明示
6. **大型ファイルは Read 全体禁止 = tail/head/grep で対象範囲のみ参照**
7. **「行末の空白文字 (trailing whitespace) 禁止」**

**追加推奨 (Loop 60-86 経験から)**:
- worker prompt に「2-commit pattern (refactor/docs commit → hash 取得 → AGENT_LOG.md / plan.md 更新 → chore commit)」を **明示的に** 記載 = 27 連続ループで完全安定動作
- 大型 file 抽出時は **precedent file の cat 全体読み込み** を明示推奨
- **worker prompt の長さに注意**: ~118-230 行で短縮効果あり (Loop 67-86 で実証 ~2.5-8 分完走)、規模 S 上限は ~200-230 行、規模 SS は ~118-150 行
- **Loop 80/85/86 precedent 適用**: worker は refactor/docs commit hash を実 hash で直接書き、chore commit hash は書かない = placeholder 残留問題予防達成 = 後継 worker も同 pattern 採用可
