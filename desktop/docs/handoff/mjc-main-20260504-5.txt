## 直前のセッションでの判断履歴と注意点

### 確立されたパターン (本セッションで定着・累積)
- **OS 互換 ErrorKind 契約強制** (本セッション ループ 1): `is_err()` のみだとエラー種別が pinned されない。`unwrap_err().kind() == ErrorKind::NotFound` に補強することで OS 文言差異を吸収しつつ type-level の不変条件を守る。
- **state machine 混在遷移テスト** (本セッション ループ 2): 同じ guard logic を持つ複数 entry point (`start` vs `start_with_output`) について「片方が active な状態で他方を呼ぶ」遷移を直接 test することで、guard 統合/分離リファクタへの耐性を獲得。
- **format prefix の層内独立確認** (本セッション ループ 4): id format `"{started_at}-{seq}"` の前半を session.rs 側で直接 test。session_store のパーサーに依存せず、format 引数順序の入れ替えを session 側で即座に検知。
- **エスケープ漏れの網羅補完** (本セッション ループ 5): 既存テスト 4 件で `_` エスケープが一度も確認されていない致命的検知漏れを補完。エスケープ対象 6 文字すべてを個別 1 文字 + escape 確認で網羅。
- **UI 文言一致の契約強制** (本セッション ループ 6): 2 つの API の error 文言が完全一致 (`"API キーが空です"`) であることを `assert_eq!(a, b)` で確認 → 体験統一性の不変条件。

### 累積パターン (前セッションから継承)
- **テスト追加方針**: private fn の **直接呼び出しテスト**で境界仕様と防御コードの存在意義を裏付ける。間接テストとの重複を許容しつつ、関数分割リファクタリングへの耐性を獲得。
- **エラー文言の「contract enforcement」パターン**: 実装と test に同じ `&'static str` を書くことで、ユーザー向け文言を不用意に変えると CI が落ちて気付ける契約強制。
- **境界値テストの「対称ペア」**: 同じ境界 (例: `MAX_JS_DATE_UNIX_SECS`) を 2 層の責任 (parser vs save wrapper) で観測。
- **3 層テスト構造** (前セッション ループ 6): 共通実装 + wrapper 経由 = 18 件で同じ仕様を 3 視点から守る (audio_utils 例)。

### 重要な技術的注意点 (踏みやすい罠)
- **`persist_if_configured` の `?` リファクタリング罠** (継承): エラーを呑み込む設計を `?` に書き換えると in-memory consistency が壊れる。test D/E (`append_keeps_segment_in_memory_when_output_dir_does_not_exist` / `finalize_returns_session_when_output_dir_does_not_exist`) が回帰検知器として機能する。
- **`classify_meeting_window_title` の whitespace-only rest 仕様**: 「`Meet -  ` (rest = whitespace のみ) は Some を返す」現挙動は docstring 未記載。
- **Teams ブラウザ版タイトル fallback は引き続き禁止** (既存方針)。
- **`validate_audio_format_properties` は cfg なし**: 同ファイルの `validate_audio_format_description` のみ `#[cfg(target_os = "macos")]` ガード。
- **wrapper fn (sanitize_sample / sanitize_pcm_sample) は削除しない**: ループ 6 のリファクタで意図的に残した。
- **session.rs の SESSION_COUNTER はグローバル**: テスト並列実行下で絶対値 (`"1000-0"` 等) を assert しない。差異 (`s1.id != s2.id`) のみ確認することで安全に保つ (本セッション ループ 4 で実装済)。
