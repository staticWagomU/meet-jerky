- worker 完走確認: **`git status --short` で対象ファイル + AGENT_LOG.md の `M` 表示確認 or `git log --oneline -3` で 2-commit 反映確認**
  - 観測時間目安: worker 起動 → 規模 SS なら ~3-8 分、規模 S なら ~5-15 分、規模 M なら ~10-15 分
  - **claude -p print mode は最終出力一括方式**: stdout は処理完了まで空のまま、AGENT_LOG.md M 表示 or git log を主観測手段にする

  - **worker 自律 2-commit pattern**: Loop 60-119 の **59 ループ連続** で観測 = 完全に安定 pattern 化
  - **worker 自律学習進化**: Loop 91-119 で worker は refactor/docs commit hash placeholder を実 hash に置換書き込み + chore commit hash は意図的に書かず + cfg(macos) module 分離の lib.rs alphabetical 挿入 + inline module 全体抽出 + alias 戦略 + frontend 軸 paradigm 移行成功 + rust 軸復帰 + 既存 file 拡張軸 = 新 paradigm + **Loop 119 で API 互換維持戦略 (pub use re-export) も自律実装成功**

## handoff 起動の制約 (重要)
- `claude-agent-handoff-main.sh` は **interactive TUI 起動のため stdin redirect 化されていない**
- そのため handoff prompt は依然として **tmux 引数長制限の影響を受ける**
- 対策 = 「ファイル参照型 handoff prompt」**precedent を継承**: 起動 prompt は短縮 + フル handoff 内容は別ファイル (`docs/handoff/mjc-main-YYYYMMDD-N.txt`) に書き、起動 prompt 内で「Read で完全に読め」と指示する (本 prompt と同パターン、連続 58 セッション目)

## worker prompt 必須要素 (100+ 連続ループで実証済、継続適用、7 つ全て明示)

worker prompt には以下を **必ず** 含める:
1. **冒頭で「AGENT_LOG.md の末尾 350 行を読め」**
2. **「AGENT_LOG.md は時系列順 = 最古ログが先頭、最新ログが末尾。新規追記は必ずファイル末尾に行う」**
3. **「先頭は絶対に触らない」**
4. **具体手順**: `tail -10 AGENT_LOG.md` で末尾を確認 → 末尾の `---` 直後に追記
5. **規模超過防止段落** = 担当範囲外の編集禁止 + test 件数の上限明示
6. **大型ファイルは Read 全体禁止 = tail/head/grep で対象範囲のみ参照**
7. **「行末の空白文字 (trailing whitespace) 禁止」**

**追加推奨 (Loop 60-119 経験から)**:
- worker prompt に「2-commit pattern (refactor/docs commit → hash 取得 → AGENT_LOG.md / plan.md 更新 → chore commit)」を **明示的に** 記載 = 59 連続ループで完全安定動作
- 大型 file 抽出時は **precedent file の cat 全体読み込み** を明示推奨
- **worker prompt の長さに注意**: ~83-230 行で短縮効果あり (Loop 67-119 で実証 ~1.5-13 分完走)
- **frontend 軸の場合**: 検証は npm run build (tsc + vite build) のみ
- **rust 軸の場合**: cargo test (704 件件数完全不変) + cargo clippy + cargo fmt + cargo build を全て worker prompt に記載
- **API 互換維持戦略 (Loop 119 precedent)**: 関数を新 file に移しつつ caller + test を全て無変更で動かすには `pub use crate::new_module::{fn1, fn2};` re-export を元 file に追加する。test の `use super::*;` import との互換性確保。
- **既存 file 拡張軸の場合 (Loop 117 precedent)**: lib.rs mod 登録は既に存在することが多い、visibility 統一 (`pub(crate)` 一致) が必要
- **責務階層精査 (Loop 119 教訓)**: paradigm 反復よりも責務階層精査を優先。低レベル parser と高レベル分類器を強引に同 file 統合すると責務肥大化リスク。
