- そのため handoff prompt は依然として **tmux 引数長制限の影響を受ける**
- 対策 = 「ファイル参照型 handoff prompt」**precedent を継承**: 起動 prompt は短縮 + フル handoff 内容は別ファイル (`docs/handoff/mjc-main-YYYYMMDD-N.txt`) に書き、起動 prompt 内で「Read で完全に読め」と指示する (本 prompt と同パターン、連続 74 セッション目)

## worker prompt 必須要素 (100+ 連続ループで実証済、継続適用、8 つ全て明示)

worker prompt には以下を **必ず** 含める:
1. **冒頭で「AGENT_LOG.md の末尾 350 行を読め」**
2. **「AGENT_LOG.md は時系列順 = 最古ログが先頭、最新ログが末尾。新規追記は必ずファイル末尾に行う」**
3. **「先頭は絶対に触らない」**
4. **具体手順**: `tail -10 AGENT_LOG.md` で末尾を確認 → 末尾の `---` 直後に追記
5. **規模超過防止段落** = 担当範囲外の編集禁止 + test 件数の上限明示
6. **大型ファイルは Read 全体禁止 = tail/head/grep で対象範囲のみ参照**
7. **「行末の空白文字 (trailing whitespace) 禁止」**
8. **「AGENT_LOG.md への append は 1 回のみ。Edit/Write を複数回繰り返さない。append が成功したら直ちに次へ進む」 (Loop 144 fault precedent から新規追加 = Loop 146-159 で 14 連続成功実証)**

**追加推奨 (Loop 60-159 経験から)**:
- worker prompt に「2-commit pattern」を **明示的に** 記載 = Loop 145-159 で 15 連続スタート安定動作
- 大型 file 抽出時は **precedent file の cat 全体読み込み** を明示推奨
- **worker prompt の長さに注意**: ~107-160 行で短縮効果あり (本セッション Loop 158 = 96 行軽量, Loop 159 = 153 行 SS-S 規模相応)
- **frontend 軸の場合**: 検証は npm run build (tsc + vite build) のみ
- **rust 軸の場合**: cargo test (704 件件数完全不変) + cargo clippy + cargo fmt + cargo build を全て worker prompt に記載
- **API 互換維持戦略 (Loop 119-159 precedent)**:
  - **rust 軸**: `pub use` re-export / `pub(crate) use` re-export = E0364 visibility 自律 fix / inline module 全体抽出時の visibility 修正 + caller fully qualified path 化 / `use` 経由で fn 抽出 + tests 移動 + 件数完全不変保証 / inline module 全体抽出 + use alias で callsite 全件無変更維持 / 重複定義削除 + use 経由参照で callsite 全件無変更維持 / **#[cfg(test)] pub(crate) fn として test build 限定公開 + 4 file から use 経由参照 (Loop 159 = test helper 集約軸 = 多 file 同時集約軸 = paradigm 6 件目)**

  - **worker 自律学習進化 paradigm 3 件目達成**: worker prompt 未指示事項を自律予測 + 対応
- **責務階層精査 (Loop 119-159 教訓)**: paradigm 反復よりも責務階層精査を優先。**既存 file 拡張軸 paradigm (Loop 145/147/149/153/157/159 = 6 件目達成)** + **新 file 作成軸 (Loop 151/155)** + **utils 間集約軸 (Loop 153 frontend 版 + Loop 157 rust 版)** + **test helper 集約軸 (Loop 159 = 5 file 同時集約軸 rust 版)**
