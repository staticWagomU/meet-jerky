- 作業: `scripts/claude-agent-start-worker.sh mjc-worker-<topic> /path/to/prompt.txt` (sonnet、stdin redirect 化 + PATH escape 化済)
- 検証: `bash scripts/agent-verify.sh <変更ファイル群>` (rust/frontend/docs を自動 skip 判定)
- コミット: `bash scripts/agent-commit.sh "メッセージ" <files>` または直接 `git add ... && git commit -m "..."`
- AGENT_LOG.md 末尾 SUMMARY 追記: ファイルへ Write → `cat <file> >> AGENT_LOG.md` で append
- watchdog: `mjc-watchdog` セッションが既に走っているのでそのまま
- canonical 名移譲: `bash scripts/agent-adopt-main.sh mjc-main-20260505-72 mjc-main`
- worker 完走確認: **`git status --short` で対象ファイル + AGENT_LOG.md の `M` 表示確認 or `git log --oneline -3` で 2-commit 反映確認**
  - 観測時間目安: 規模 SSS = ~2-3 分、規模 SS = ~3-5 分、規模 SS-S = ~4-7 分、規模 S = ~5-15 分
  - **本セッション Loop 144 例**: 規模 SS = ~240 秒で worker fault (二重書き) → メイン直接救済で完走
  - **本セッション Loop 145 例**: 規模 SS = ~200 秒完走 (worker prompt ~190 行 = 重複削除 + 既存 file 拡張軸 paradigm 詳細記載で長め + worker 自律学習進化 paradigm 2 件目発展)
  - **worker 自律 2-commit pattern**: Loop 60-143 の **84 ループ連続** で観測 → Loop 144 fault でリセット → Loop 145 で再開 = 1 連続スタート
  - **worker 自律学習進化** (Loop 143/145 で 2 件確立): worker prompt 未指示事項を自律予測 + 対応 = build error 未然防止 / callsite grep 自律置換

## handoff 起動の制約 (重要)
- `claude-agent-handoff-main.sh` は **interactive TUI 起動のため stdin redirect 化されていない**
- そのため handoff prompt は依然として **tmux 引数長制限の影響を受ける**
- 対策 = 「ファイル参照型 handoff prompt」**precedent を継承**: 起動 prompt は短縮 + フル handoff 内容は別ファイル (`docs/handoff/mjc-main-YYYYMMDD-N.txt`) に書き、起動 prompt 内で「Read で完全に読め」と指示する (本 prompt と同パターン、連続 67 セッション目)

## worker prompt 必須要素 (100+ 連続ループで実証済、継続適用、8 つ全て明示)

worker prompt には以下を **必ず** 含める:
1. **冒頭で「AGENT_LOG.md の末尾 350 行を読め」**
2. **「AGENT_LOG.md は時系列順 = 最古ログが先頭、最新ログが末尾。新規追記は必ずファイル末尾に行う」**
3. **「先頭は絶対に触らない」**
4. **具体手順**: `tail -10 AGENT_LOG.md` で末尾を確認 → 末尾の `---` 直後に追記
5. **規模超過防止段落** = 担当範囲外の編集禁止 + test 件数の上限明示
6. **大型ファイルは Read 全体禁止 = tail/head/grep で対象範囲のみ参照**
7. **「行末の空白文字 (trailing whitespace) 禁止」**
8. **「AGENT_LOG.md への append は 1 回のみ。Edit/Write を複数回繰り返さない。append が成功したら直ちに次へ進む」 (Loop 144 fault precedent から新規追加)**

**追加推奨 (Loop 60-145 経験から)**:
- worker prompt に「2-commit pattern」を **明示的に** 記載 = 84 連続ループで完全安定動作 (Loop 144 fault でリセット → Loop 145 で再開)
- 大型 file 抽出時は **precedent file の cat 全体読み込み** を明示推奨
- **worker prompt の長さに注意**: ~140-190 行で短縮効果あり (Loop 67-145 で実証 ~60-300 秒完走)
- **frontend 軸の場合**: 検証は npm run build (tsc + vite build) のみ
- **rust 軸の場合**: cargo test (704 件件数完全不変) + cargo clippy + cargo fmt + cargo build を全て worker prompt に記載
- **API 互換維持戦略 (Loop 119-145 precedent)**:
  - **rust 軸**: `pub use` re-export (Loop 119) / `pub(crate) use` re-export = E0364 visibility 自律 fix (Loop 121) / inline module 全体抽出時の visibility 修正 + caller 1 件 fully qualified path 化で完結 (Loop 123)
  - **frontend 軸**: 純粋関数 export 化 + import 1 行追加で完結 (Loop 125-143 標準 paradigm) / **新 file が他の utils file から定数/型/関数 import = utils 内 cross-import paradigm (Loop 129/133/135/141/143 で 2-6 件目確立 = paradigm 進化継続)** / **既存 file 拡張軸 = 新 file 作成なし + 重複削除 + import 1 行追加 (Loop 145 = paradigm 新規確立)**
  - **worker 自律学習進化 paradigm**: worker prompt 未指示事項を自律予測 + 対応 = TranscriptionEngineType 不要 import 削除 (Loop 143) / MicrophoneSection / SystemAudioSection callsite 自律置換 (Loop 145) = paradigm 進化発展継続
- **責務階層精査 (Loop 119-145 教訓)**: paradigm 反復よりも責務階層精査を優先。**既存 file 拡張軸 paradigm (Loop 145)**: 同 entity + 同 tier の重複候補発見時は新 file ではなく既存 file 拡張軸採用 = DRY 集約
